# Раздел 4: Модели и Model I/O — сердце LLM-приложений

## Введение: Модель как чёрный ящик с характером

В центре любого LLM-приложения находится языковая модель. Всё остальное — промпты, парсеры, инструменты, цепочки — это инфраструктура вокруг неё. Модель принимает текст (или сообщения), обрабатывает их своими миллиардами параметров и выдаёт ответ. Казалось бы, простой интерфейс: вход → обработка → выход.

Но эта простота обманчива. За фасадом API скрывается сложнейшая система со множеством настроек, влияющих на результат. Temperature определяет "креативность" ответа. Top-p ограничивает выборку токенов. Max tokens лимитирует длину. Stop sequences останавливают генерацию. И это лишь основные параметры.

Более того, разные модели имеют разные API. OpenAI использует один формат запросов, Anthropic — другой, локальные модели через Ollama — третий. Каждый провайдер добавляет свои уникальные параметры, ограничения, особенности аутентификации.

LangChain решает эту проблему через унификацию. Все модели — облачные и локальные, от разных провайдеров, с разными параметрами — реализуют единый интерфейс. Это позволяет писать код, который не зависит от конкретного провайдера, и переключаться между моделями изменением одной строки.

## Часть 1: Два мира моделей — LLM и ChatModel

### Историческое разделение

Когда LangChain создавался в конце 2022 года, существовало чёткое разделение между двумя типами моделей.

**LLM (Large Language Model)** — классические модели, обученные предсказывать следующий токен. Они принимают строку текста и возвращают строку-продолжение. GPT-3 (без "turbo") был таким: вы даёте ему начало текста, он продолжает. Нет понятия "системное сообщение", нет ролей — просто текст.

**ChatModel** — модели, обученные на диалогах. Они принимают список сообщений с ролями (system, user, assistant) и возвращают сообщение. ChatGPT, Claude, Gemini — все современные модели работают так.

В LangChain это разделение отражалось в разных базовых классах: `BaseLLM` для completion-моделей и `BaseChatModel` для chat-моделей. Каждый имел свой интерфейс: LLM принимал строку, ChatModel — список сообщений.

### Современная реальность

К 2024-2025 годам баланс сместился полностью в сторону chat-моделей. OpenAI даже deprecate'нула свой completions API для последних моделей. Практически все современные модели — chat-based.

LangChain адаптировался. Хотя классы LLM всё ещё существуют для совместимости, рекомендуется использовать ChatModel везде. Это избавляет от путаницы и делает код консистентным.

```python
# Современный подход — использовать ChatModel
from langchain_openai import ChatOpenAI

model = ChatOpenAI(model="gpt-4")
result = model.invoke("Привет!")  # Строка автоматически конвертируется в HumanMessage
```

Даже если вы передаёте строку, LangChain автоматически оборачивает её в HumanMessage. Это удобно для простых случаев, хотя для продакшн-кода лучше быть явным:

```python
from langchain_core.messages import HumanMessage

result = model.invoke([HumanMessage(content="Привет!")])
```

## Часть 2: Провайдеры и интеграции

### Экосистема провайдеров

LangChain поддерживает десятки провайдеров моделей. Каждый вынесен в отдельный пакет:

```bash
pip install langchain-openai       # OpenAI, Azure OpenAI
pip install langchain-anthropic    # Anthropic Claude
pip install langchain-google-genai # Google Gemini
pip install langchain-mistralai    # Mistral
pip install langchain-groq         # Groq (быстрый inference)
pip install langchain-ollama       # Локальные модели через Ollama
pip install langchain-huggingface  # HuggingFace модели
pip install langchain-gigachat     # GigaChat
```

Каждый пакет устанавливает только необходимые зависимости для работы с конкретным провайдером. Не нужно устанавливать SDK всех провайдеров, если вы используете только OpenAI.

### OpenAI: стандарт де-факто

OpenAI — самый популярный провайдер, и langchain-openai — самая используемая интеграция.

```python
from langchain_openai import ChatOpenAI

model = ChatOpenAI(
    model="gpt-5-mini",
    temperature=0.7,
    max_tokens=1000,
    api_key="sk-..."  # Или через переменную OPENAI_API_KEY
)
```

Azure OpenAI использует тот же пакет, но другой класс:

```python
from langchain_openai import AzureChatOpenAI

model = AzureChatOpenAI(
    azure_deployment="my-gpt4-deployment",
    azure_endpoint="https://my-resource.openai.azure.com/",
    api_version="2024-02-01"
)
```

### Anthropic: конкурент с характером

```python
from langchain_anthropic import ChatAnthropic

model = ChatAnthropic(
    model="claude-4-5-sonnet-{num}",
    temperature=0.5,
    max_tokens=4096
)
```

### Локальные модели: контроль и приватность

Для приватности данных или экономии на API-вызовах можно запускать модели локально. Ollama — самый удобный способ:

```python
from langchain_ollama import ChatOllama

model = ChatOllama(
    model="llama3.2:8b",
    temperature=0.7
)
```

Популярные локальные модели:
- `qwen` — китайская модель, хорошо понимает код
- `mistral` — европейская альтернатива
- `gemma` — модель от Google
- `llama3.2` — открытая модель от Meta
- `glm`, `nemotron`, etc

Локальные модели требуют GPU для комфортной работы. На CPU скорость генерации падает в десятки раз.

## Часть 3: Параметры генерации

### Temperature: креативность vs детерминизм

Temperature — главный параметр, влияющий на "случайность" ответа. Математически он масштабирует логиты (сырые предсказания модели) перед применением softmax.

При **temperature = 0** модель всегда выбирает наиболее вероятный токен. Ответы детерминированы: один и тот же вход даёт один и тот же выход. Идеально для задач, где нужна консистентность: классификация, извлечение данных, следование шаблону.

При **temperature = 1** распределение вероятностей используется как есть. Модель может выбрать менее вероятные токены, создавая вариативность. Хорошо для творческих задач: генерация текста, брейнсторминг.

При **temperature > 1** распределение "сглаживается", редкие токены становятся более вероятными. Ответы становятся непредсказуемыми, иногда бессвязными. Используется редко.

```python
# Для классификации — детерминированные ответы
classifier = ChatOpenAI(model="gpt-5-mini", temperature=0)

# Для творческого письма — вариативность
writer = ChatOpenAI(model="gpt-5-mini", temperature=0.8)
```

Практический совет: начните с temperature=0 для предсказуемости, увеличивайте только если нужна вариативность.

### Top-P (Nucleus Sampling)

Top-P — альтернативный способ контроля случайности. Вместо масштабирования логитов, он ограничивает выборку токенов, чья совокупная вероятность составляет p.

При **top_p = 0.9** модель рассматривает только токены, составляющие 90% вероятностной массы. Редкие токены (с очень низкой вероятностью) отсекаются.

При **top_p = 0.1** выборка очень узкая — только самые вероятные токены.

```python
model = ChatOpenAI(
    model="gpt-5-mini",
    temperature=0.7,
    top_p=0.9
)
```

OpenAI рекомендует изменять либо temperature, либо top_p, но не оба одновременно. Комбинирование может дать непредсказуемые результаты.

### Max Tokens: ограничение длины

Max tokens ограничивает длину генерируемого ответа. Это защита от "болтливости" модели и контроль затрат.

```python
# Короткие ответы для классификации
model = ChatOpenAI(model="gpt-5-mini", max_tokens=50)

# Длинные ответы для генерации статей
model = ChatOpenAI(model="gpt-5-mini", max_tokens=4000)
```

Важно: max_tokens ограничивает только выход, не вход.

### Stop Sequences: контролируемая остановка

Stop sequences — строки, при генерации которых модель останавливается. Это мощный инструмент для структурированного вывода.

```python
model = ChatOpenAI(
    model="gpt-5-mini",
    stop=["```", "\n\nВопрос:"]
)
```

Если модель генерирует "```", она остановится. Это полезно, когда вы просите модель написать код в блоке и хотите избежать лишнего текста после.

### Seed: воспроизводимость

Для отладки и тестирования полезно получать одинаковые ответы на одинаковые запросы. Параметр seed фиксирует случайность (при temperature > 0):

```python
model = ChatOpenAI(
    model="gpt-5-mini",
    temperature=0.5,
    seed=42
)
```

С одним и тем же seed ответы будут идентичны (или очень похожи — провайдеры не гарантируют 100% воспроизводимость).

## Часть 4: Отслеживание использования

### Подсчёт токенов

Токены — единица измерения для LLM. Вы платите за токены, контекстное окно измеряется в токенах, лимиты rate limiting обычно в токенах в минуту.

LangChain предоставляет инструменты для подсчёта:

```python
from langchain_openai import ChatOpenAI

model = ChatOpenAI(model="gpt-5-mini")
result = model.invoke("Расскажи о Python")

print(result.usage_metadata)
# {'input_tokens': 12, 'output_tokens': 150, 'total_tokens': 162}
```

Поле `usage_metadata` появляется после вызова и содержит информацию об использованных токенах.

Для предварительной оценки (до вызова) можно использовать токенизатор:

```python
from langchain_openai import ChatOpenAI

model = ChatOpenAI(model="gpt-5-mini")
num_tokens = model.get_num_tokens("Это текст для подсчёта токенов")
print(f"Токенов: {num_tokens}")
```

### Callbacks для мониторинга

Для систематического отслеживания используйте callbacks:

```python
from langchain_core.callbacks import BaseCallbackHandler

class UsageTracker(BaseCallbackHandler):
    def __init__(self):
        self.total_tokens = 0
        self.total_cost = 0

    def on_llm_end(self, response, **kwargs):
        usage = response.llm_output.get("token_usage", {})
        tokens = usage.get("total_tokens", 0)
        self.total_tokens += tokens
        # Примерный расчёт стоимости
        self.total_cost += tokens * 0.00001

tracker = UsageTracker()
model = ChatOpenAI(model="gpt-5-mini", callbacks=[tracker])

# После нескольких вызовов
print(f"Всего токенов: {tracker.total_tokens}")
print(f"Примерная стоимость: ${tracker.total_cost:.4f}")
```

### Экономика токенов

Понимание стоимости токенов критично для production-систем. Выходные токены дороже входных в 3-5 раз. Это важно учитывать при проектировании: длинный промпт с коротким ответом дешевле, чем короткий промпт с длинным ответом.

Для минимизации затрат:

- Используйте дешёвые модели для простых задач (классификация, извлечение данных)
- Ограничивайте max_tokens где возможно
- Кэшируйте ответы на повторяющиеся запросы
- Батчите запросы для оптимизации накладных расходов

## Часть 5: Мультимодальность

### Работа с изображениями

Современные модели (GPT, Claude, Gemini) понимают изображения. LangChain поддерживает передачу изображений через специальный формат сообщений:

```python
from langchain_core.messages import HumanMessage

message = HumanMessage(
    content=[
        {"type": "text", "text": "Что на этом изображении?"},
        {
            "type": "image_url",
            "image_url": {"url": "https://example.com/image.jpg"}
        }
    ]
)

result = model.invoke([message])
```

Для локальных изображений используйте base64-кодирование:

```python
import base64

def image_to_base64(path: str) -> str:
    with open(path, "rb") as f:
        return base64.b64encode(f.read()).decode()

message = HumanMessage(
    content=[
        {"type": "text", "text": "Опиши эту диаграмму"},
        {
            "type": "image_url",
            "image_url": {
                "url": f"data:image/png;base64,{image_to_base64('diagram.png')}"
            }
        }
    ]
)
```

### Особенности работы с Vision

Изображения "стоят" токены. Большое изображение может занимать тысячи токенов контекста. OpenAI автоматически масштабирует изображения, но всё равно учитывайте это при планировании.

Некоторые модели поддерживают параметр детализации:

```python
{
    "type": "image_url",
    "image_url": {
        "url": "...",
        "detail": "low"  # "low", "high", или "auto"
    }
}
```

"Low" использует меньше токенов, но модель видит меньше деталей. "High" — больше токенов, лучше распознавание мелких элементов.

## Часть 6: Потоковая генерация

### Стриминг токенов

Метод `stream` возвращает итератор по токенам:

```python
for chunk in model.stream("Расскажи историю"):
    print(chunk.content, end="", flush=True)
```

Каждый chunk — это AIMessageChunk с несколькими токенами. Пользователь видит ответ по мере генерации, а не после многосекундного ожидания.

### Асинхронный стриминг

Для веб-приложений используйте асинхронную версию:

```python
async for chunk in model.astream("Расскажи историю"):
    yield chunk.content  # Передаём клиенту через SSE
```

### Стриминг через цепочки

Когда модель часть цепочки, стриминг работает прозрачно:

```python
chain = prompt | model | parser

# Если parser поддерживает стриминг, токены пройдут через него
for chunk in chain.stream({"topic": "AI"}):
    print(chunk, end="")
```

Но не все парсеры поддерживают стриминг. JsonOutputParser, например, ждёт полный ответ, потому что частичный JSON невалиден. StrOutputParser стримит прозрачно.

### Стриминг событий

Для полного контроля используйте `astream_events`:

```python
async for event in chain.astream_events(input, version="v2"):
    if event["event"] == "on_chat_model_stream":
        print(event["data"]["chunk"].content, end="")
    elif event["event"] == "on_chain_end":
        print("\n[Завершено]")
```

Это позволяет отслеживать все этапы выполнения, а не только финальный вывод.

## Часть 7: Интеграция с LCEL

### Модель как Runnable

Все chat-модели реализуют Runnable и интегрируются в LCEL:

```python
chain = prompt | model | parser
```

Модель принимает сообщения от промпта, генерирует ответ, передаёт парсеру. Всё работает с invoke, batch, stream, их асинхронными версиями.

### Bind: привязка параметров

Метод `bind` создаёт версию модели с привязанными параметрами:

```python
model = ChatOpenAI(model="gpt-5.2")

# Модель для генерации JSON
json_model = model.bind(response_format={"type": "json_object"})

# Модель со стоп-последовательностью
code_model = model.bind(stop=["```"])
```

Это полезно, когда одна и та же базовая модель используется с разными настройками в разных частях приложения.

### Configurable: динамические параметры

Для параметров, определяемых в runtime:

```python
from langchain_core.runnables import ConfigurableField

model = ChatOpenAI(model="gpt-5-mini").configurable_fields(
    temperature=ConfigurableField(
        id="temperature",
        name="Temperature",
        description="Креативность ответов"
    )
)

# Использование с разной температурой
result1 = model.invoke("...", config={"configurable": {"temperature": 0}})
result2 = model.invoke("...", config={"configurable": {"temperature": 0.9}})
```

### Configurable Alternatives: переключение моделей

Для A/B тестирования или fallback-логики:

```python
model = ChatOpenAI(model="gpt-5-mini").configurable_alternatives(
    ConfigurableField(id="model_provider"),
    default_key="openai",
    anthropic=ChatAnthropic(model="claude-3-5-sonnet-20241022"),
    local=ChatOllama(model="llama3.2:8b")
)

# Переключение провайдера в runtime
result = model.invoke(
    "...",
    config={"configurable": {"model_provider": "anthropic"}}
)
```

## Часть 8: Best Practices

### Выбор модели под задачу

Не все задачи требуют GPT-5. Часто дешёвая модель справляется так же хорошо:

| Задача | Рекомендуемая модель |
|--------|---------------------|
| Классификация | GPT-5-mini, Claude Haiku |
| Извлечение данных | GPT-5-mini |
| Генерация контента | GPT-5.2, Claude Sonnet |
| Сложные рассуждения | GPT-5.2, Claude Opus |
| Анализ изображений | GPT-5.2, Claude Sonnet |
| Кодирование | GPT-5.2, Claude Sonnet |

Начинайте с дешёвой модели, переходите на дорогую только если качество недостаточно.

### Обработка ошибок

Вызовы API могут падать. Всегда используйте retry и fallback:

```python
model = (
    ChatOpenAI(model="gpt-5-mini")
    .with_retry(stop_after_attempt=3)
    .with_fallbacks([ChatOpenAI(model="gpt-5.2")])
)
```

### Rate Limiting

Провайдеры ограничивают количество запросов. При батчинге ограничивайте параллелизм:

```python
results = model.batch(
    inputs,
    config={"max_concurrency": 5}  # Максимум 5 параллельных запросов
)
```

### Логирование и мониторинг

В production обязательно логируйте:

- Входные данные (для отладки)
- Выходные данные (для анализа качества)
- Использование токенов (для биллинга)
- Время ответа (для оптимизации)
- Ошибки

## Заключение: Модель как строительный блок

Языковая модель — это фундамент, на котором строится всё остальное. LangChain абстрагирует различия между провайдерами, предоставляя единый интерфейс. Вы можете начать с OpenAI, переключиться на Anthropic для длинного контекста, добавить локальную модель для приватных данных — и всё это с минимальными изменениями кода.

Понимание параметров генерации (temperature, top_p, max_tokens) позволяет тонко настраивать поведение модели под конкретные задачи. Отслеживание токенов и затрат критично для production-систем.

## Ключевые термины

| Термин | Определение |
|--------|-------------|
| **ChatModel** | Модель, работающая с сообщениями (system, user, assistant) |
| **Temperature** | Параметр, контролирующий случайность генерации (0 = детерминизм, 1 = стандартно) |
| **Top-P** | Альтернативный способ контроля случайности через ограничение вероятностной массы |
| **Max Tokens** | Ограничение длины генерируемого ответа |
| **Stop Sequences** | Строки, при генерации которых модель останавливается |
| **Seed** | Параметр для воспроизводимости результатов |
| **Token** | Единица текста для LLM (примерно 4 символа английского текста) |
