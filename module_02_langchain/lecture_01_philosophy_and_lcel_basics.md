# Раздел 1: Философия LangChain и основы LCEL

## Введение: Почему нам нужна прослойка между кодом и моделью

Представьте себе, что вы — опытный программист, который только что получил доступ к API самой мощной языковой модели в мире. Вы открываете документацию, находите endpoint для chat completions, пишете простой HTTP-запрос, отправляете его — и получаете ответ. Магия! Ваш первый AI-ассистент заработал за пять минут.

Проходит неделя. Ваш прототип превращается в нечто более сложное. Теперь вам нужно сохранять историю диалога. Потом добавить возможность поиска по документам. Затем — интеграцию с внешними API. Ещё через месяц вы обнаруживаете, что половина вашего кода — это не бизнес-логика, а "клей" между различными компонентами: парсинг JSON-ответов модели, обработка ошибок при сетевых сбоях, управление контекстным окном, логирование запросов для отладки.

И вот вы смотрите на свой код и понимаете: вы написали собственный мини-фреймворк. Причём написали его на ходу, без продуманной архитектуры, с десятками неочевидных решений, которые теперь разбросаны по всей кодовой базе. Каждый раз, когда нужно добавить нового провайдера моделей или изменить логику обработки, приходится копаться в этом хаосе.

Именно эту проблему решает LangChain. Это не магическая библиотека, которая делает AI "умнее". Это инженерный инструмент, который предоставляет продуманные абстракции для типовых задач при работе с языковыми моделями. Подобно тому, как веб-фреймворки вроде Django или Flask освобождают разработчика от необходимости вручную парсить HTTP-заголовки и управлять сессиями, LangChain берёт на себя рутину оркестрации LLM-приложений.

## Часть 1: Рождение и эволюция LangChain

### Краткая история: от хобби-проекта до стандарта индустрии

История LangChain началась в октябре 2022 года, буквально через несколько недель после того, как OpenAI открыла публичный доступ к API ChatGPT. Харрисон Чейз, инженер с опытом работы в стартапах машинного обучения, начал экспериментировать с объединением языковых моделей и внешних инструментов. Его первоначальная идея была проста: что если дать модели возможность не просто отвечать на вопросы, а выполнять действия — искать информацию в Google, обращаться к базам данных, запускать код?

Первые версии LangChain были набором Python-скриптов, которые Чейз выложил на GitHub. Проект взорвался. За три месяца он собрал более 10 000 звёзд — невиданная скорость роста даже для эпохи AI-хайпа. К началу 2023 года LangChain привлёк венчурное финансирование и превратился в полноценную компанию.

Почему LangChain стал настолько популярным? Причин несколько. Во-первых, он появился в идеальный момент — когда тысячи разработчиков одновременно начали экспериментировать с GPT и столкнулись с одними и теми же проблемами. Во-вторых, проект развивался с невероятной скоростью, добавляя поддержку новых моделей и интеграций буквально каждый день. В-третьих, LangChain предложил удобный ментальный фреймворк для думания о LLM-приложениях: цепочки, агенты, инструменты, память.

Однако бурный рост имел и обратную сторону. Ранние версии LangChain критиковали за избыточную сложность, неинтуитивные абстракции и постоянно ломающийся API. Каждую неделю выходила новая версия, которая делала предыдущий код нерабочим. Документация не успевала за изменениями. Некоторые разработчики в шутку называли LangChain "самой хорошо документированной библиотекой с самой устаревшей документацией".

К 2024-2025 годам ситуация стабилизировалась. Команда LangChain провела масштабный рефакторинг, выделив ядро библиотеки (langchain-core) и представив новую декларативную парадигму — LangChain Expression Language (LCEL). Это тот LangChain, который мы будем изучать в этом модуле.

### Архитектура современного LangChain

Современный LangChain — это не монолитная библиотека, а экосистема взаимосвязанных пакетов. Понимание её структуры критически важно для эффективной работы.

Ядром всего является пакет `langchain-core`. Здесь определены базовые абстракции: интерфейс Runnable, схемы сообщений, система промптов. Этот пакет намеренно минималистичен — он не содержит интеграций с конкретными провайдерами моделей или внешними сервисами. Его задача — определить "язык", на котором говорят все остальные компоненты.

Интеграции с провайдерами моделей вынесены в отдельные пакеты: `langchain-openai`, `langchain-anthropic`, `langchain-google-genai` и десятки других. Такое разделение имеет практический смысл: вам не нужно устанавливать зависимости для всех возможных провайдеров, если вы работаете только с одним.

Пакет `langchain` (без суффикса) содержит высокоуровневые конструкции: готовые цепочки, агенты, инструменты для работы с документами. Это "строительные блоки", из которых можно быстро собрать типовые приложения.

Отдельно стоит упомянуть `langchain-community` — огромную коллекцию интеграций, созданных сообществом: загрузчики документов для сотен форматов, коннекторы к базам данных, интеграции с SaaS-сервисами. Качество этих интеграций неоднородно, но их разнообразие впечатляет.

Наконец, `langgraph` — относительно новый пакет для построения циклических графов состояний, которому мы посвятим отдельный модуль курса. LangGraph решает ограничения линейных цепочек и позволяет создавать по-настоящему сложных агентов.

### Философский фундамент: композиция как принцип

Если попытаться выразить философию LangChain одним словом, это слово — "композиция". Вся архитектура построена на идее, что сложные системы должны собираться из простых, переиспользуемых компонентов.

Эта идея не нова. В функциональном программировании композиция функций — базовый принцип. Unix-философия "делай одно дело и делай его хорошо" + "соединяй программы через pipes" — это тоже композиция. React-компоненты, которые вкладываются друг в друга — композиция. LangChain применяет ту же идею к LLM-приложениям.

Рассмотрим простейший пример. У вас есть три компонента: шаблон промпта, который форматирует пользовательский вопрос; языковая модель, которая генерирует ответ; парсер, который извлекает структурированные данные из ответа. Каждый компонент выполняет одну задачу. Но вместе они образуют конвейер: вопрос проходит через шаблон, результат шаблона — через модель, ответ модели — через парсер.

В императивном стиле это выглядело бы как последовательность вызовов функций с промежуточными переменными. В LangChain это записывается как `prompt | model | parser` — буквально конвейер, где вертикальная черта означает "передай результат дальше". Эта простота записи — не просто синтаксический сахар. Она отражает глубинную структуру: каждый компонент — это преобразователь, который принимает вход и производит выход, не заботясь о том, откуда пришёл вход и куда уйдёт выход.

## Часть 2: Интерфейс Runnable — сердце LangChain

### Что такое Runnable и зачем он нужен

Если LangChain — это королевство, то интерфейс Runnable — его конституция. Каждый компонент в LangChain, будь то модель, промпт, парсер или цепочка, реализует этот интерфейс. Понимание Runnable — ключ к пониманию всей библиотеки.

Runnable — это контракт, который гарантирует, что любой компонент можно вызвать стандартным способом. Интерфейс определяет несколько методов, главные из которых — `invoke`, `batch`, `stream` и их асинхронные аналоги `ainvoke`, `abatch`, `astream`.

Метод `invoke` — базовый способ вызова. Вы передаёте вход, получаете выход. Просто, понятно, синхронно. Это ваш рабочий вариант для отладки и простых сценариев.

Метод `batch` позволяет обработать несколько входов за один вызов. Зачем это нужно? Представьте, что вам нужно классифицировать тысячу документов. Вызывать `invoke` в цикле тысячу раз — неэффективно. Каждый вызов — это отдельный HTTP-запрос к API провайдера с соответствующими накладными расходами. Метод `batch` может оптимизировать это: группировать запросы, использовать параллельные соединения, минимизировать латентность.

Метод `stream` — для случаев, когда важна отзывчивость интерфейса. Языковые модели генерируют текст токен за токеном. Если ждать полного ответа, пользователь видит пустой экран секунды или даже десятки секунд. Стриминг позволяет показывать ответ по мере генерации, как это делает ChatGPT. Психологически это воспринимается гораздо лучше, даже если общее время ответа то же самое.

Асинхронные версии этих методов (с префиксом "a") критически важны для продакшн-приложений. Python-сервер, обрабатывающий сотни одновременных запросов, не может позволить себе блокироваться на каждом вызове к внешнему API. Асинхронность позволяет эффективно использовать ресурсы: пока один запрос ждёт ответа от OpenAI, сервер обрабатывает другие запросы.

### Унификация: почему всё — Runnable

Красота интерфейса Runnable в его универсальности. Посмотрите на эти разные компоненты:

Языковая модель (ChatOpenAI) — это Runnable. Вход: список сообщений. Выход: сообщение-ответ.

Шаблон промпта (ChatPromptTemplate) — это Runnable. Вход: словарь с переменными. Выход: форматированные сообщения.

Парсер (StrOutputParser) — это Runnable. Вход: сообщение модели. Выход: строка.

Цепочка из трёх компонентов — тоже Runnable! Вход: словарь с переменными. Выход: строка.

Это означает, что цепочку можно использовать как компонент внутри другой цепочки. Можно батчить цепочки. Можно стримить цепочки. Все методы, которые работают с простыми компонентами, работают и с композициями любой сложности.

Такой подход называется "единообразие" (uniformity) или "рекурсивная композиция". Он мощнее, чем кажется на первый взгляд. В традиционном императивном коде, когда вы хотите добавить, скажем, логирование ко всем вызовам, приходится модифицировать каждую функцию. В LangChain можно создать Runnable-обёртку для логирования и применить её к любому компоненту или цепочке — логирование автоматически применится ко всем внутренним вызовам.

### Практический пример: анатомия вызова

Давайте разберём, что происходит, когда вы вызываете простую цепочку. Предположим, у нас есть:

```python
chain = prompt | model | parser
result = chain.invoke({"topic": "квантовые компьютеры"})
```

Шаг первый: метод `invoke` цепочки получает словарь `{"topic": "квантовые компьютеры"}`. Цепочка знает, что она состоит из трёх компонентов, соединённых последовательно.

Шаг второй: цепочка вызывает `prompt.invoke({"topic": "квантовые компьютеры"})`. Шаблон промпта подставляет значение переменной и возвращает объект ChatPromptValue, содержащий список отформатированных сообщений.

Шаг третий: цепочка вызывает `model.invoke(chat_prompt_value)`. Модель извлекает сообщения из ChatPromptValue, формирует HTTP-запрос к API OpenAI, ждёт ответа, парсит JSON, оборачивает результат в объект AIMessage.

Шаг четвёртый: цепочка вызывает `parser.invoke(ai_message)`. Парсер извлекает текстовое содержимое из AIMessage и возвращает строку.

Шаг пятый: цепочка возвращает эту строку как финальный результат.

Обратите внимание: на каждом шаге выход предыдущего компонента автоматически становится входом следующего. Преобразования типов (словарь → ChatPromptValue → AIMessage → строка) происходят прозрачно. Вам не нужно вручную извлекать поля, конвертировать форматы, обрабатывать промежуточные результаты.

## Часть 3: LangChain Expression Language — декларативный подход

### Синтаксис Pipe и его корни

LangChain Expression Language (LCEL) — это декларативный язык для описания потоков данных через компоненты LangChain. Ключевой элемент его синтаксиса — оператор `|`, который соединяет компоненты в цепочку.

Если вы работали с командной строкой Unix/Linux, этот синтаксис покажется знакомым. Команда `cat file.txt | grep "error" | wc -l` передаёт содержимое файла в grep для фильтрации, затем результат — в wc для подсчёта строк. Каждая программа читает данные из stdin, обрабатывает их и пишет в stdout. Pipe (`|`) соединяет stdout одной программы с stdin следующей.

LCEL работает аналогично. Каждый Runnable-компонент имеет определённый тип входа и тип выхода. Pipe соединяет выход левого компонента со входом правого. Результат — новый Runnable, который можно соединять дальше.

Но есть и важное отличие от Unix pipes. В Unix данные передаются как байтовые потоки — программы должны сами парсить и форматировать текст. В LCEL передаются структурированные Python-объекты. Более того, LangChain автоматически выполняет необходимые преобразования типов. Если промпт возвращает ChatPromptValue, а модель ожидает список сообщений, преобразование происходит прозрачно.

### RunnableSequence: что скрывается за оператором Pipe

Когда вы пишете `a | b | c`, Python вызывает специальный метод `__or__`, определённый в классе Runnable. Этот метод создаёт новый объект класса RunnableSequence, который хранит список компонентов и умеет последовательно вызывать их.

RunnableSequence — это тоже Runnable. Поэтому результат выражения `a | b` можно сохранить в переменную и использовать как обычный компонент: передавать в функции, комбинировать с другими компонентами, вызывать invoke/batch/stream.

```python
# Эти записи эквивалентны:
chain = prompt | model | parser
chain = RunnableSequence(first=prompt, middle=[model], last=parser)
```

Первый вариант — синтаксический сахар для второго. Но какой элегантный сахар! Сравните читаемость. Pipe-нотация буквально показывает поток данных слева направо. Вам не нужно разбираться в том, какой аргумент куда передаётся — структура очевидна из самой записи.

### RunnableParallel: параллельное ветвление

Последовательное выполнение — не единственный паттерн. Часто нужно выполнить несколько операций параллельно и собрать результаты. Для этого существует RunnableParallel.

Представьте задачу: получив вопрос пользователя, вы хотите одновременно найти релевантные документы и сгенерировать уточняющие вопросы. Эти операции независимы — их можно выполнять параллельно. RunnableParallel принимает словарь, где ключи — это имена результатов, а значения — Runnable-компоненты:

```python
parallel = RunnableParallel({
    "documents": retriever,
    "follow_up_questions": question_generator
})
```

При вызове `parallel.invoke(user_question)` оба компонента получают один и тот же вход (вопрос пользователя) и выполняются параллельно. Результат — словарь с ключами "documents" и "follow_up_questions", содержащий выходы соответствующих компонентов.

В LCEL есть короткий синтаксис для RunnableParallel — просто передайте словарь в качестве компонента цепочки:

```python
chain = {"context": retriever, "question": RunnablePassthrough()} | prompt | model
```

Здесь первый элемент цепочки — словарь, который автоматически оборачивается в RunnableParallel. Retriever получает вход и возвращает документы, а RunnablePassthrough просто пропускает вход без изменений. Результат — словарь с ключами "context" и "question", который передаётся в промпт.

### RunnablePassthrough: искусство ничегонеделания

RunnablePassthrough — самый простой компонент в LangChain. Он принимает вход и возвращает его без изменений. Зачем нужен компонент, который ничего не делает?

Ответ кроется в параллельном выполнении. Когда RunnableParallel распределяет вход по нескольким веткам, иногда одна из веток должна просто "пронести" вход дальше без трансформации. RunnablePassthrough — это способ сказать "возьми вход как есть".

Другой случай — когда нужно добавить к входу дополнительные данные. Метод `RunnablePassthrough.assign()` создаёт компонент, который копирует все поля входного словаря и добавляет новые:

```python
chain = RunnablePassthrough.assign(
    current_time=lambda _: datetime.now().isoformat()
)
# Вход: {"question": "Какая погода?"}
# Выход: {"question": "Какая погода?", "current_time": "2025-01-24T12:30:00"}
```

Это элегантный способ обогащения данных в середине цепочки без разрыва потока.

### RunnableLambda: интеграция произвольного кода

Не всё можно выразить стандартными компонентами LangChain. Иногда нужно вставить в цепочку произвольную Python-функцию: преобразовать формат данных, вызвать внешний API, выполнить валидацию. RunnableLambda оборачивает обычную функцию в Runnable-интерфейс.

```python
def extract_first_sentence(text: str) -> str:
    return text.split('.')[0] + '.'

first_sentence = RunnableLambda(extract_first_sentence)
chain = prompt | model | parser | first_sentence
```

RunnableLambda можно создать и с помощью декоратора `@chain`:

```python
from langchain_core.runnables import chain

@chain
def extract_first_sentence(text: str) -> str:
    return text.split('.')[0] + '.'
```

Важное замечание: RunnableLambda может оборачивать как синхронные, так и асинхронные функции. Если вы передаёте асинхронную функцию, она будет правильно вызываться при использовании ainvoke/astream.

## Часть 4: Поток данных и трансформации типов

### Схема входов и выходов

Каждый Runnable имеет определённую схему входа и выхода. Для простых компонентов это обычно очевидно: модель принимает сообщения и возвращает сообщение, парсер принимает сообщение и возвращает строку. Но при построении сложных цепочек полезно явно понимать, какие типы данных текут через конвейер.

LangChain предоставляет методы для инспекции схем: `input_schema` и `output_schema` возвращают Pydantic-модели, описывающие структуру данных. Метод `get_graph()` возвращает граф выполнения, который можно визуализировать.

```python
chain = prompt | model | parser
print(chain.input_schema.schema())  # Схема входного словаря
print(chain.output_schema.schema())  # Схема выхода
chain.get_graph().draw_mermaid()  # ASCII-арт схема
```

Эти инструменты неоценимы при отладке. Если цепочка выдаёт ошибку "unexpected input type", инспекция схем поможет понять, какой компонент ожидает какой формат.

### Автоматическое преобразование типов

LangChain старается быть "умным" в отношении типов. Если промпт-шаблон возвращает ChatPromptValue, а модель ожидает список сообщений, преобразование происходит автоматически. Если вы передаёте строку туда, где ожидается HumanMessage, строка автоматически оборачивается в HumanMessage.

Эта магия работает благодаря системе "coercion" (приведения типов), встроенной в базовые классы. Каждый Runnable умеет преобразовывать входные данные в ожидаемый формат, если преобразование однозначно.

Однако магия имеет пределы. Если преобразование неоднозначно или невозможно, вы получите ошибку. Поэтому при отладке сложных цепочек полезно явно проверять типы на каждом шаге.

### Обработка ошибок в цепочках

Что происходит, когда компонент в середине цепочки выбрасывает исключение? По умолчанию — исключение всплывает наверх, и вся цепочка падает. Это логично: если не удалось получить ответ от модели, бессмысленно пытаться его парсить.

Но иногда нужна более тонкая обработка ошибок. Метод `with_fallbacks()` позволяет указать резервные компоненты:

```python
model_with_fallback = main_model.with_fallbacks([backup_model])
```

Если основная модель выбрасывает исключение, LangChain автоматически попробует резервную. Это особенно полезно при работе с ненадёжными API: если OpenAI недоступен, можно переключиться на Anthropic или локальную модель.

Метод `with_retry()` добавляет автоматические повторные попытки с экспоненциальной задержкой — стандартный паттерн для борьбы с временными сбоями сети.

## Часть 5: Практические паттерны LCEL

### Паттерн "Ветвление и слияние"

Один из самых мощных паттернов LCEL — ветвление потока на параллельные ветки с последующим слиянием результатов. Типичный пример — RAG (Retrieval-Augmented Generation), где нужно одновременно получить контекст из базы знаний и сохранить оригинальный вопрос.

```python
chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | prompt
    | model
    | parser
)
```

Первый элемент цепочки — RunnableParallel (записанный как словарь). Он создаёт два параллельных потока: retriever ищет релевантные документы, RunnablePassthrough пропускает вопрос без изменений. Результат — словарь с ключами "context" и "question". Этот словарь передаётся в prompt-шаблон, который использует оба значения для форматирования.

### Паттерн "Условная маршрутизация"

Иногда логика приложения требует условного ветвления: разные входы должны обрабатываться по-разному. RunnableBranch позволяет определить несколько условий и соответствующих им обработчиков.

```python
from langchain_core.runnables import RunnableBranch

branch = RunnableBranch(
    (lambda x: x["type"] == "question", question_chain),
    (lambda x: x["type"] == "command", command_chain),
    default_chain  # Если ни одно условие не сработало
)
```

RunnableBranch проверяет условия по порядку и выполняет первый подходящий обработчик. Это аналог конструкции if-elif-else, но в декларативной форме.

### Паттерн "Аккумуляция контекста"

При построении сложных цепочек часто нужно накапливать информацию по мере выполнения. Например, сначала классифицировать запрос, затем на основе классификации выбрать стратегию обработки, затем сгенерировать ответ с учётом и классификации, и результатов обработки.

RunnablePassthrough.assign() идеально подходит для этого:

```python
chain = (
    RunnablePassthrough.assign(classification=classify_chain)
    | RunnablePassthrough.assign(search_results=lambda x: search(x["classification"]))
    | prompt
    | model
)
```

На каждом шаге к исходному словарю добавляются новые поля. К моменту достижения промпта словарь содержит и оригинальный запрос, и классификацию, и результаты поиска.

## Часть 6: Визуализация и отладка

### Инспекция графа выполнения

Сложные LCEL-цепочки могут быть трудны для понимания. К счастью, LangChain предоставляет инструменты визуализации.

Метод `get_graph()` возвращает объект Graph, представляющий структуру цепочки. У этого объекта есть методы для различных форматов вывода:

```python
chain.get_graph().draw_mermaid()  # Текстовая диаграмма в формате Mermaid
chain.get_graph().draw_ascii()    # ASCII-арт
```

Mermaid-диаграммы можно рендерить в Jupyter-ноутбуках или в онлайн-редакторах. Они показывают узлы (компоненты) и рёбра (потоки данных), помогая понять структуру даже очень сложных цепочек.

### Трейсинг выполнения

Визуализация графа показывает структуру цепочки, но не показывает, что происходит во время выполнения. Для этого существует трейсинг — механизм записи всех вызовов с их входами, выходами и временем выполнения.

LangSmith — облачная платформа от создателей LangChain — предоставляет интерактивный интерфейс для просмотра трейсов. Но для локальной разработки достаточно включить вывод в консоль:

```python
from langchain_core.tracers import ConsoleCallbackHandler

chain.invoke(input, config={"callbacks": [ConsoleCallbackHandler()]})
```

Трейсинг неоценим при отладке. Он показывает, какой компонент получил какой вход, что вернул, сколько времени занял вызов. Если цепочка ведёт себя неожиданно, трейс обычно сразу показывает, где проблема.

## Заключение: Мышление композициями

Ключевая идея, которую важно унести с собой: LangChain — это не магия, а набор продуманных абстракций. Интерфейс Runnable унифицирует все компоненты. Pipe-оператор соединяет их в последовательности. RunnableParallel создаёт параллельные ветки. RunnableLambda интегрирует произвольный код.

Эти примитивы — как кубики LEGO. По отдельности каждый прост. Но комбинируя их, можно построить системы произвольной сложности. При этом каждая композиция сама является Runnable и может использоваться как кубик для ещё более сложных конструкций.

## Ключевые термины

| Термин | Определение |
|--------|-------------|
| **Runnable** | Базовый интерфейс LangChain, определяющий методы invoke, batch, stream для всех компонентов |
| **LCEL** | LangChain Expression Language — декларативный синтаксис для построения цепочек через pipe-оператор |
| **RunnableSequence** | Последовательность компонентов, выполняемых один за другим |
| **RunnableParallel** | Параллельное выполнение нескольких компонентов с общим входом |
| **RunnablePassthrough** | Компонент, передающий вход без изменений |
| **RunnableLambda** | Обёртка для интеграции произвольных Python-функций в LCEL |
| **Pipe-оператор** | Символ `\|`, соединяющий компоненты в последовательность |
