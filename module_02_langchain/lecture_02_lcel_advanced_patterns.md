# Раздел 2: LangChain Expression Language — продвинутые паттерны

## Введение: От простых цепочек к реальным системам

Реальные LLM-приложения редко бывают простыми линейными цепочками. Они содержат условную логику: разные типы запросов обрабатываются по-разному. Они должны быть устойчивы к сбоям: если один провайдер моделей недоступен, нужно переключиться на другой. Они работают с потоковыми данными: пользователь видит ответ по мере его генерации, а не после многосекундного ожидания. Они должны быть эффективны: кэширование, батчинг, параллельное выполнение.

Все эти возможности встроены в LCEL. Нужно лишь знать, как их использовать.

## Часть 1: Динамическая маршрутизация

### Проблема ветвления

Представьте чат-бота для службы поддержки. Пользователь может задать технический вопрос, пожаловаться на качество обслуживания, спросить о статусе заказа или просто поболтать. Каждый тип запроса требует разной обработки: технические вопросы нужно сопоставлять с базой знаний, жалобы — эскалировать живому оператору, статус заказа — проверять через API системы логистики.

Как реализовать такое ветвление в LCEL? У нас есть несколько инструментов, каждый со своими сильными сторонами.

### RunnableBranch: явные условия

RunnableBranch — самый прямолинейный способ условного ветвления. Вы определяете список пар "условие → обработчик", и LangChain выполняет первый обработчик, условие которого истинно.

```python
from langchain_core.runnables import RunnableBranch

router = RunnableBranch(
    (lambda x: x["category"] == "technical", technical_chain),
    (lambda x: x["category"] == "complaint", complaint_chain),
    (lambda x: x["category"] == "order_status", order_chain),
    default_chain
)
```

Обратите внимание на последний элемент без условия — это обработчик по умолчанию, который срабатывает, если ни одно условие не подошло. Если default не указан и ни одно условие не сработало, RunnableBranch выбросит исключение.

Условия — это функции, принимающие вход и возвращающие булево значение. Они могут быть сколь угодно сложными: проверять несколько полей, вызывать внешние сервисы, использовать модели машинного обучения для классификации.

Важный нюанс: условия проверяются последовательно, сверху вниз. Первое истинное условие "побеждает", остальные не проверяются. Это значит, что порядок условий имеет значение, и более специфичные условия должны идти раньше более общих.

### RunnableRouter: маршрутизация по ключу

Когда маршрутов много, а условия сводятся к проверке одного поля, RunnableBranch становится громоздким. RunnableRouter предлагает альтернативу — маршрутизацию по значению ключа.

```python
from langchain_core.runnables import RouterRunnable

routes = {
    "technical": technical_chain,
    "complaint": complaint_chain,
    "order_status": order_chain,
    "general": general_chain
}

router = RouterRunnable(routes, key="category")
```

Здесь `key="category"` означает, что RouterRunnable будет смотреть на значение поля "category" во входном словаре и выбирать соответствующую цепочку. Если значение "technical", выполняется technical_chain. Если "complaint" — complaint_chain. И так далее.

Это чище и понятнее, когда маршрутизация действительно сводится к switch-case по одному полю.

### Семантическая маршрутизация: когда логику определяет модель

Но откуда берётся поле "category"? В предыдущих примерах мы предполагали, что классификация уже выполнена. В реальности первый шаг — сам определить категорию.

Здесь появляется паттерн "семантическая маршрутизация". Вместо жёстко закодированных правил мы используем языковую модель для классификации.

```python
classification_prompt = ChatPromptTemplate.from_template("""
Классифицируй следующий запрос пользователя.
Возможные категории: technical, complaint, order_status, general.
Верни только название категории, без объяснений.

Запрос: {query}
Категория:""")

classification_chain = classification_prompt | model | StrOutputParser()

full_chain = (
    RunnablePassthrough.assign(category=classification_chain)
    | router
)
```

Сначала выполняется classification_chain, который определяет категорию. Результат добавляется к входу через RunnablePassthrough.assign(). Затем router использует эту категорию для выбора обработчика.

Красота этого подхода в гибкости. Не нужно вручную определять ключевые слова для каждой категории. Модель понимает контекст, улавливает нюансы формулировок, справляется с опечатками и сленгом. А когда появляются новые категории, достаточно обновить промпт — не нужно переписывать логику парсинга.

### Сложные решения: multi-step routing

Иногда маршрутизация требует нескольких шагов. Например, сначала определяем, это вопрос или команда. Если вопрос — определяем область (технический, юридический, финансовый). Если команда — определяем тип действия (создать, удалить, изменить).

LCEL позволяет строить такие деревья решений композицией:

```python
first_level = RunnableBranch(
    (is_question, question_router),
    (is_command, command_router),
    fallback_chain
)

question_router = RunnableBranch(
    (is_technical, technical_qa_chain),
    (is_legal, legal_qa_chain),
    general_qa_chain
)

command_router = RunnableBranch(
    (is_create, create_handler),
    (is_delete, delete_handler),
    update_handler
)
```

Каждый уровень ветвления — отдельный RunnableBranch. Они вкладываются друг в друга, образуя дерево. Входные данные спускаются по дереву до листового обработчика, который и производит финальный ответ.

## Часть 2: Устойчивость к сбоям

### Реальность распределённых систем

LLM-приложение — это распределённая система. Ваш код обращается к внешним API: OpenAI, Anthropic, векторным базам данных, поисковым движкам. Каждый из этих сервисов может быть временно недоступен, перегружен, медленно отвечать или возвращать ошибки.

Без правильной обработки сбоев ваше приложение падает при первой же проблеме с внешним сервисом. Пользователь видит непонятную ошибку, теряет доверие к продукту, уходит к конкурентам. В худшем случае незавершённые транзакции оставляют систему в неконсистентном состоянии.

LCEL предоставляет два основных механизма устойчивости: повторные попытки (retry) и резервные варианты (fallback).

### with_retry: автоматические повторные попытки

Метод `with_retry()` добавляет к любому Runnable логику автоматического повтора при сбоях. Простейший вариант:

```python
robust_model = model.with_retry()
```

Если вызов модели выбрасывает исключение (таймаут, сетевая ошибка, rate limiting), LangChain автоматически повторит запрос. По умолчанию используется экспоненциальная задержка: 1 секунда, 2 секунды, 4 секунды... Это даёт перегруженному сервису время восстановиться.

Параметры можно настроить:

```python
robust_model = model.with_retry(
    stop_after_attempt=5,       # Максимум 5 попыток
    wait_exponential_jitter=True,  # Добавить случайный разброс к задержке
    retry_if_exception_type=(TimeoutError, ConnectionError)  # Какие ошибки повторять
)
```

Последний параметр особенно важен. Не все ошибки стоит повторять. Rate limiting? Да, подождём и попробуем снова. Ошибка валидации входных данных? Нет смысла — запрос сформирован неправильно, повтор не поможет. Ошибка авторизации? Нет — API-ключ недействителен.

### with_fallbacks: резервные варианты

Повторные попытки помогают с временными сбоями, но что если сервис недоступен продолжительное время? Или если один провайдер дороже другого и вы хотите использовать дешёвый вариант, переключаясь на дорогой только при необходимости?

Метод `with_fallbacks()` определяет цепочку резервных вариантов:

```python
resilient_model = primary_model.with_fallbacks([
    secondary_model,
    tertiary_model
])
```

Если primary_model выбрасывает исключение, LangChain пробует secondary_model. Если и он падает — tertiary_model. Только если все варианты исчерпаны, исключение всплывает наверх.

Типичные сценарии использования fallbacks:

Мультипровайдерная устойчивость. Основная модель — GPT-4 от OpenAI. Резервная — Claude от Anthropic. Если OpenAI недоступен, переключаемся на Claude. Пользователь может заметить разницу в стиле ответов, но лучше немного другой ответ, чем ошибка.

```python
model = ChatOpenAI(model="gpt-4").with_fallbacks([
    ChatAnthropic(model="claude-3-sonnet"),
    ChatOpenAI(model="gpt-3.5-turbo")  # Дешёвая модель как последний resort
])
```

Экономия затрат. Сначала пробуем локальную модель (бесплатно, но менее качественно). Если она не справляется, переключаемся на облачную. "Не справляется" здесь может означать слишком низкую уверенность ответа или специфический тип ошибки.

Graceful degradation. Основная цепочка сложная: RAG с векторным поиском, реранкинг, мультишаговая генерация. Резервная цепочка простая: прямой вызов модели без контекста. Если сложная система падает (например, векторная БД недоступна), пользователь получает хотя бы какой-то ответ.

### Комбинирование retry и fallbacks

Эти механизмы ортогональны и хорошо комбинируются:

```python
resilient_model = (
    ChatOpenAI(model="gpt-4")
    .with_retry(stop_after_attempt=3)
    .with_fallbacks([
        ChatAnthropic(model="claude-3-sonnet").with_retry(stop_after_attempt=2),
        ChatOpenAI(model="gpt-3.5-turbo").with_retry(stop_after_attempt=2)
    ])
)
```

Логика такая: пробуем GPT-4 до трёх раз. Если все три попытки неудачны, переключаемся на Claude (до двух попыток). Если Claude тоже падает — на GPT-3.5 (до двух попыток). Итого до семи попыток прежде чем сдаться.

### Обработка исключений в цепочках

Что если ошибка происходит не в модели, а в другом компоненте цепочки — например, в парсере? По умолчанию исключение всплывает и прерывает всю цепочку. Но иногда нужна более тонкая обработка.

RunnableLambda позволяет обернуть часть цепочки в try-except:

```python
@chain
def safe_parse(response):
    try:
        return json_parser.invoke(response)
    except OutputParserException:
        return {"error": "Не удалось распарсить ответ", "raw": response.content}

safe_chain = prompt | model | safe_parse
```

Теперь вместо падения всей цепочки при ошибке парсинга, мы получаем словарь с информацией об ошибке и сырым ответом модели. Приложение может показать пользователю сообщение "Произошла ошибка, попробуйте переформулировать вопрос" вместо технического stack trace.

## Часть 3: Стриминг — потоковая передача данных

### Почему стриминг важен

Когда вы используете ChatGPT, ответ появляется слово за словом. Это не просто красивый эффект — это принципиально улучшает пользовательский опыт.

Представьте два сценария. В первом вы нажимаете кнопку "Отправить" и смотрите на пустой экран 10 секунд. Потом появляется ответ. В втором вы нажимаете кнопку, и через полсекунды начинают появляться слова. Через 10 секунд ответ полностью готов.

Объективно время до полного ответа одинаковое. Субъективно второй сценарий воспринимается как гораздо более быстрый и отзывчивый. Пользователь видит, что система работает, начинает читать ответ, не чувствует дискомфорта от ожидания.

Языковые модели генерируют текст токен за токеном. Каждый токен доступен сразу после генерации — не нужно ждать полного ответа. Стриминг позволяет передавать эти токены клиенту по мере их появления.

### Метод stream: основы

Любой Runnable поддерживает метод `stream()`, который возвращает итератор по частям ответа:

```python
for chunk in chain.stream({"question": "Что такое LangChain?"}):
    print(chunk, end="", flush=True)
```

Для языковых моделей каждый chunk — это фрагмент текста (обычно несколько токенов). Для других компонентов логика может отличаться.

Асинхронная версия использует async for:

```python
async for chunk in chain.astream({"question": "Что такое LangChain?"}):
    print(chunk, end="", flush=True)
```

### Стриминг через цепочки

Когда вы стримите из цепочки, не все компоненты одинаково "стримят". Некоторые компоненты потоковые по своей природе — они производят выход порциями. Другие — блокирующие: они ждут полного входа, обрабатывают его и выдают полный выход.

Языковые модели потоковые: они генерируют токен за токеном. Промпт-шаблоны блокирующие: им нужны все переменные, чтобы сформировать сообщение. Многие парсеры блокирующие: им нужен полный ответ модели для извлечения структуры.

LangChain оптимизирует поток автоматически. Если цепочка `prompt | model | parser`, где парсер блокирующий, стриминг всё равно имеет смысл. Токены модели будут накапливаться по мере генерации, и как только модель завершит ответ, парсер обработает его. Пользователь увидит токены модели в реальном времени, если вы используете стриминг событий (об этом ниже).

### Стриминг событий: полный контроль

Метод `stream()` возвращает только финальный выход цепочки. Но иногда нужно видеть промежуточные шаги: что сделал retriever, какой ответ дала модель до парсинга, сколько времени занял каждый компонент.

Метод `astream_events()` предоставляет полный лог событий:

```python
async for event in chain.astream_events({"question": "..."}, version="v2"):
    kind = event["event"]
    if kind == "on_chat_model_stream":
        print(event["data"]["chunk"].content, end="")
    elif kind == "on_chain_end":
        print(f"\nЦепочка завершена за {event['data']['output']}")
```

События включают:

- `on_chain_start` / `on_chain_end` — начало и конец выполнения цепочки
- `on_llm_start` / `on_llm_end` — вызов языковой модели
- `on_chat_model_stream` — каждый токен от chat-модели
- `on_retriever_start` / `on_retriever_end` — поиск в векторной базе
- `on_tool_start` / `on_tool_end` — вызов инструмента

Каждое событие содержит метаданные: имя компонента, теги, время, входные и выходные данные. Это мощнейший инструмент для отладки и мониторинга.

## Часть 4: Эффективность — батчинг и конкурентность

### Батчинг: обработка группами

Метод `batch()` обрабатывает несколько входов за один вызов. Зачем это нужно, когда можно просто вызвать `invoke()` в цикле?

Ответ — эффективность. Каждый HTTP-запрос имеет накладные расходы: установка соединения, передача заголовков, ожидание ответа. Если у вас 100 запросов и каждый занимает 100мс чистой обработки плюс 50мс накладных расходов, последовательное выполнение займёт 100 × 150мс = 15 секунд. Батчинг может уменьшить накладные расходы.

Более того, некоторые провайдеры API поддерживают батчевые endpoint'ы, которые обрабатывают несколько запросов за один вызов. LangChain может использовать их автоматически.

```python
questions = [
    {"topic": "квантовые компьютеры"},
    {"topic": "нейронные сети"},
    {"topic": "блокчейн"}
]

results = chain.batch(questions)
```

Параметр `max_concurrency` ограничивает количество параллельных запросов:

```python
results = chain.batch(questions, config={"max_concurrency": 5})
```

Это важно для соблюдения rate limits провайдеров. OpenAI, например, ограничивает количество запросов в минуту. Бесконтрольный параллелизм приведёт к ошибкам rate limiting.

### Асинхронность и конкурентность

Батчинг работает и с асинхронными вызовами:

```python
results = await chain.abatch(questions, config={"max_concurrency": 5})
```

Асинхронность особенно важна в серверных приложениях. Синхронный код блокирует поток на время ожидания ответа от API. Если у вас 10 потоков и 100 одновременных пользователей, 90 из них ждут.

Асинхронный код использует event loop: пока один запрос ждёт ответа от OpenAI, другие запросы обрабатываются. Тот же сервер может обслуживать тысячи одновременных соединений.

### Конкурентность в RunnableParallel

RunnableParallel выполняет ветки параллельно. Но "параллельно" в Python не означает "одновременно" — GIL ограничивает параллелизм CPU-bound операций. Однако для I/O-bound операций (а вызовы API именно такие) асинхронность даёт реальный параллелизм.

```python
parallel = RunnableParallel({
    "summary": summarize_chain,
    "keywords": keyword_chain,
    "sentiment": sentiment_chain
})

# Все три цепочки выполняются конкурентно
result = await parallel.ainvoke(document)
```

Если каждая цепочка занимает 2 секунды, последовательное выполнение заняло бы 6 секунд. Параллельное выполнение — около 2 секунд (плюс небольшие накладные расходы).

## Часть 5: Конфигурация времени выполнения

### Зачем нужна runtime-конфигурация

До сих пор мы создавали компоненты с фиксированными параметрами: модель с определённой temperature, цепочка с фиксированным промптом. Но иногда параметры должны определяться в момент вызова.

Примеры:

- Разные пользователи имеют разные API-ключи
- Temperature зависит от типа задачи (креативная vs фактологическая)
- Таймауты зависят от приоритета запроса
- Логирование включается для отладки конкретных пользователей

### RunnableConfig: механизм передачи конфигурации

Каждый метод Runnable (`invoke`, `batch`, `stream`) принимает необязательный параметр `config`:

```python
result = chain.invoke(
    {"question": "..."},
    config={"callbacks": [my_callback], "tags": ["production"]}
)
```

Конфигурация передаётся через всю цепочку — каждый компонент получает её и может использовать.

Стандартные поля конфигурации:

- `callbacks` — список callback-обработчиков для трейсинга
- `tags` — теги для группировки логов
- `metadata` — произвольные метаданные
- `run_name` — имя прогона для отладки
- `max_concurrency` — ограничение параллелизма
- `configurable` — пользовательские настраиваемые параметры

### Configurable Fields: параметризуемые компоненты

LangChain позволяет сделать параметры компонента настраиваемыми в runtime:

```python
model = ChatOpenAI(temperature=0.7).configurable_fields(
    temperature=ConfigurableField(
        id="temperature",
        name="Креативность",
        description="Насколько креативными будут ответы"
    )
)

# Использование с разной температурой
creative_result = model.invoke(prompt, config={"configurable": {"temperature": 0.9}})
factual_result = model.invoke(prompt, config={"configurable": {"temperature": 0.0}})
```

Метод `configurable_fields()` оборачивает компонент в ConfigurableRunnable. При вызове он смотрит в `config["configurable"]` и подставляет значения.

### Configurable Alternatives: переключение компонентов

Ещё более мощный механизм — configurable alternatives. Он позволяет переключаться между разными компонентами в runtime:

```python
model = ChatOpenAI(model="gpt-4").configurable_alternatives(
    ConfigurableField(id="model"),
    default_key="gpt4",
    claude=ChatAnthropic(model="claude-3-sonnet"),
    gpt35=ChatOpenAI(model="gpt-3.5-turbo")
)

# Использование разных моделей
result1 = model.invoke(prompt)  # GPT-4 (по умолчанию)
result2 = model.invoke(prompt, config={"configurable": {"model": "claude"}})
result3 = model.invoke(prompt, config={"configurable": {"model": "gpt35"}})
```

Это мощный инструмент для A/B тестирования, экспериментов с моделями, и предоставления пользователям выбора.

## Часть 6: Callbacks и наблюдаемость

### Система callbacks

Callbacks — это хуки, которые вызываются на различных этапах выполнения. Они позволяют "подслушивать" внутреннюю работу цепочки без изменения её логики.

```python
from langchain_core.callbacks import BaseCallbackHandler

class LoggingHandler(BaseCallbackHandler):
    def on_llm_start(self, serialized, prompts, **kwargs):
        print(f"LLM started with {len(prompts)} prompts")

    def on_llm_end(self, response, **kwargs):
        print(f"LLM finished: {response.generations[0][0].text[:50]}...")

    def on_chain_error(self, error, **kwargs):
        print(f"Error occurred: {error}")

handler = LoggingHandler()
result = chain.invoke(input, config={"callbacks": [handler]})
```

Доступные callback-методы покрывают все этапы: start/end для LLM, chain, tool, retriever, плюс специальные методы для стриминга и ошибок.

### Встроенные handlers

LangChain предоставляет несколько готовых обработчиков:

ConsoleCallbackHandler — выводит все события в консоль. Отлично для отладки в development.

StreamingStdOutCallbackHandler — стримит токены модели в stdout. Простой способ увидеть стриминг без написания кода.

FileCallbackHandler — записывает события в файл. Для офлайн-анализа.

```python
from langchain_core.tracers import ConsoleCallbackHandler

result = chain.invoke(input, config={"callbacks": [ConsoleCallbackHandler()]})
```

## Часть 7: Кэширование

### Зачем кэшировать ответы LLM

Вызовы языковых моделей дороги — и по времени, и по деньгам. Если пользователи задают похожие вопросы, имеет смысл сохранять ответы и использовать их повторно.

LangChain поддерживает несколько стратегий кэширования.

### InMemoryCache: простейший вариант

```python
from langchain.cache import InMemoryCache
from langchain.globals import set_llm_cache

set_llm_cache(InMemoryCache())
```

Теперь повторные вызовы с идентичными входами возвращают кэшированный результат мгновенно. Кэш живёт в памяти процесса и теряется при перезапуске.

### SQLiteCache: персистентный кэш

```python
from langchain.cache import SQLiteCache

set_llm_cache(SQLiteCache(database_path="llm_cache.db"))
```

Кэш сохраняется в SQLite-файл и переживает перезапуски. Хорошо для development и небольших приложений.

### Redis и другие бэкенды

Для продакшн-систем с несколькими инстансами приложения нужен разделяемый кэш:

```python
from langchain.cache import RedisCache
import redis

redis_client = redis.Redis.from_url("redis://localhost:6379")
set_llm_cache(RedisCache(redis_client))
```

### Semantic cache: кэширование по смыслу

Обычный кэш работает по точному совпадению входа. Но "Что такое Python?" и "Расскажи о Python" — по сути один вопрос.

Семантический кэш использует эмбеддинги для поиска похожих вопросов:

```python
from langchain.cache import RedisSemanticCache

set_llm_cache(RedisSemanticCache(
    redis_url="redis://localhost:6379",
    embedding=OpenAIEmbeddings()
))
```

Если новый вопрос семантически близок к кэшированному (cosine similarity выше порога), возвращается кэшированный ответ.

Это мощно, но требует осторожности. Слишком агрессивное семантическое кэширование может возвращать нерелевантные ответы на вопросы, которые кажутся похожими, но отличаются в важных деталях.

## Заключение: Собираем всё вместе

Ключевой принцип: LCEL декларативен. Вместо императивного кода с if-else, try-except, threading вы описываете структуру обработки. LangChain берёт на себя детали реализации: как именно параллелизовать, как передавать стрим, как обрабатывать ошибки.

Это не означает, что LangChain скрывает сложность. Скорее, он канализирует её в понятные абстракции. Понимание этих абстракций — ключ к эффективной работе с библиотекой.
