# Раздел 3: Управление промптами — искусство разговора с моделью

## Введение: Промпт как интерфейс

Если языковая модель — это мозг AI-системы, то промпт — это интерфейс к этому мозгу. Качество промпта часто определяет разницу между системой, которая впечатляет, и системой, которая разочаровывает. Вы можете использовать самую мощную модель в мире, но если ваш промпт сформулирован неудачно, результат будет посредственным.

В первом модуле касались prompt engineering как дисциплину: Chain of Thought, Few-shot prompting, ReAct. Это были паттерны рассуждений — способы структурировать мышление модели. Сегодня мы смотрим на промпты с инженерной точки зрения: как организовать, параметризовать и переиспользовать промпты в контексте LangChain.

LangChain предоставляет мощную систему шаблонов промптов. Это не просто string formatting с подстановкой переменных. Это типизированные объекты, которые интегрируются с остальной экосистемой, поддерживают композицию, валидацию, частичное заполнение и многое другое.

## Часть 1: Анатомия сообщений

### Парадигма сообщений в chat-моделях

Современные языковые модели, особенно chat-модели вроде GPT-5 или Claude, работают не с сырым текстом, а со списком сообщений. Каждое сообщение имеет роль, определяющую его функцию в диалоге.

Роль **System** — это инструкции для модели, определяющие её поведение, персону, ограничения. System-сообщение обычно идёт первым и задаёт контекст для всего диалога. Модели обучены придавать системным сообщениям особый вес — они влияют на тон, стиль и содержание всех последующих ответов.

Роль **Human** (или User) — это сообщения от пользователя. Вопросы, команды, входные данные — всё, что приходит от человека.

Роль **AI** (или Assistant) — это ответы модели. В контексте истории диалога это предыдущие ответы, которые модель должна учитывать для поддержания связности.

Роль **Function** или **Tool** — относительно новая, связанная с функциональным вызовом. Это результаты выполнения инструментов, которые возвращаются модели для формирования финального ответа.

Понимание ролей критично для эффективного prompt engineering. System-сообщение — это место для метаинструкций, которые модель должна соблюдать всегда. Human-сообщение — для конкретного запроса. AI-сообщение в начале диалога может "затравить" стиль ответа (если вы хотите, чтобы модель всегда начинала с определённой фразы).

### Классы сообщений в LangChain

LangChain представляет сообщения как объекты с типизацией:

```python
from langchain_core.messages import SystemMessage, HumanMessage, AIMessage

messages = [
    SystemMessage(content="Ты — полезный ассистент, отвечающий кратко."),
    HumanMessage(content="Что такое Python?"),
    AIMessage(content="Python — высокоуровневый язык программирования."),
    HumanMessage(content="А для чего он используется?")
]
```

Каждый класс сообщения наследуется от BaseMessage и содержит:

- `content` — основное содержимое (строка или список для мультимодальных сообщений)
- `role` — роль, выведенная из типа класса
- `additional_kwargs` — дополнительные параметры (например, для function calling)
- `id` — опциональный идентификатор

Типизация важна. Она позволяет LangChain и моделям правильно обрабатывать сообщения. При отправке в OpenAI API объекты сериализуются в формат, который API ожидает. При получении ответа — десериализуются обратно в объекты.

### Мультимодальные сообщения

Современные модели умеют работать не только с текстом, но и с изображениями. LangChain поддерживает мультимодальный контент:

```python
from langchain_core.messages import HumanMessage

message = HumanMessage(
    content=[
        {"type": "text", "text": "Что изображено на этой картинке?"},
        {"type": "image_url", "image_url": {"url": "https://example.com/image.jpg"}}
    ]
)
```

Здесь `content` — не строка, а список элементов. Каждый элемент указывает свой тип и данные. Модель получает и текст, и изображение, и может отвечать на вопросы о картинке.

Можно также передавать изображения в формате base64, что полезно для локальных файлов или сгенерированных изображений:

```python
import base64

with open("local_image.jpg", "rb") as f:
    image_data = base64.b64encode(f.read()).decode()

message = HumanMessage(
    content=[
        {"type": "text", "text": "Опиши это изображение"},
        {"type": "image_url", "image_url": {"url": f"data:image/jpeg;base64,{image_data}"}}
    ]
)
```

## Часть 2: ChatPromptTemplate — сердце системы промптов

### Создание шаблонов

ChatPromptTemplate — основной класс для создания промптов в LangChain. Он описывает структуру диалога с placeholder'ами для динамических данных.

Простейший способ создания — метод `from_messages`:

```python
from langchain_core.prompts import ChatPromptTemplate

prompt = ChatPromptTemplate.from_messages([
    ("system", "Ты — эксперт в области {domain}. Отвечай подробно и точно."),
    ("human", "{question}")
])
```

Здесь кортежи `(роль, шаблон)` определяют сообщения. Фигурные скобки — placeholder'ы для переменных. При вызове `prompt.invoke({"domain": "физики", "question": "Что такое энтропия?"})` переменные подставляются, и получается список сообщений.

Можно также использовать классы сообщений напрямую:

```python
from langchain_core.prompts import ChatPromptTemplate, SystemMessagePromptTemplate, HumanMessagePromptTemplate

prompt = ChatPromptTemplate.from_messages([
    SystemMessagePromptTemplate.from_template("Ты — эксперт в области {domain}."),
    HumanMessagePromptTemplate.from_template("{question}")
])
```

Результат тот же, но более явный. Этот стиль полезен, когда нужна дополнительная настройка отдельных сообщений.

### Шаблоны с примерами (Few-shot)

Few-shot prompting — техника, когда вы показываете модели несколько примеров желаемого поведения перед запросом. LangChain упрощает создание few-shot промптов:

```python
examples = [
    {"input": "Я люблю этот продукт!", "output": "позитивный"},
    {"input": "Ужасное качество, разочарован.", "output": "негативный"},
    {"input": "Нормально, ничего особенного.", "output": "нейтральный"}
]

prompt = ChatPromptTemplate.from_messages([
    ("system", "Определи тональность текста. Отвечай одним словом: позитивный, негативный или нейтральный."),
    *[
        msg
        for ex in examples
        for msg in [("human", ex["input"]), ("ai", ex["output"])]
    ],
    ("human", "{text}")
])
```

Здесь примеры разворачиваются в последовательность human-ai пар, имитируя предыдущий диалог. Модель "видит", что на похожие запросы она отвечала определённым образом, и следует этому паттерну.

Для более сложных случаев существует FewShotChatMessagePromptTemplate:

```python
from langchain_core.prompts import FewShotChatMessagePromptTemplate

example_prompt = ChatPromptTemplate.from_messages([
    ("human", "{input}"),
    ("ai", "{output}")
])

few_shot_prompt = FewShotChatMessagePromptTemplate(
    examples=examples,
    example_prompt=example_prompt
)

final_prompt = ChatPromptTemplate.from_messages([
    ("system", "Ты — классификатор тональности."),
    few_shot_prompt,
    ("human", "{text}")
])
```

FewShotChatMessagePromptTemplate сам управляет форматированием примеров, что делает код чище и позволяет легко менять набор примеров.

### Динамический выбор примеров

В реальных приложениях набор примеров не статичен. Если у вас тысячи примеров, включать их все в промпт непрактично — не хватит контекстного окна. Нужен механизм выбора релевантных примеров.

LangChain предоставляет селекторы примеров. Самый мощный — SemanticSimilarityExampleSelector, который выбирает примеры, семантически близкие к входному запросу:

```python
from langchain_core.example_selectors import SemanticSimilarityExampleSelector
from langchain_openai import OpenAIEmbeddings
from langchain_community.vectorstores import Chroma

example_selector = SemanticSimilarityExampleSelector.from_examples(
    examples,
    OpenAIEmbeddings(),
    Chroma,
    k=3  # Выбрать 3 наиболее похожих примера
)

dynamic_prompt = FewShotChatMessagePromptTemplate(
    example_selector=example_selector,
    example_prompt=example_prompt
)
```

Теперь при каждом вызове промпта selector сравнивает входной текст с примерами и выбирает наиболее релевантные. Если пользователь спрашивает о продукте, примеры будут о продуктах. Если о сервисе — о сервисе.

## Часть 3: MessagesPlaceholder — динамическая вставка истории

### Проблема истории диалога

Chat-бот должен помнить предыдущие сообщения. Пользователь говорит "Он хороший", а бот должен понимать, что "он" — это продукт, обсуждавшийся ранее. Для этого история диалога передаётся модели вместе с новым запросом.

Но история — это переменное количество сообщений. В отличие от фиксированных placeholder'ов вроде `{question}`, количество сообщений истории заранее неизвестно. MessagesPlaceholder решает эту проблему.

```python
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

prompt = ChatPromptTemplate.from_messages([
    ("system", "Ты — дружелюбный ассистент."),
    MessagesPlaceholder(variable_name="history"),
    ("human", "{input}")
])
```

При вызове в `history` передаётся список сообщений любой длины:

```python
history = [
    HumanMessage(content="Привет!"),
    AIMessage(content="Привет! Чем могу помочь?"),
    HumanMessage(content="Расскажи о Python"),
    AIMessage(content="Python — это язык программирования...")
]

result = prompt.invoke({"history": history, "input": "А о его библиотеках?"})
```

Результат — промпт с системным сообщением, всей историей посередине, и новым вопросом пользователя в конце.

### Optional placeholders

Иногда история может быть пустой (первый запрос в диалоге). По умолчанию MessagesPlaceholder требует, чтобы переменная была предоставлена. Параметр `optional=True` делает placeholder необязательным:

```python
prompt = ChatPromptTemplate.from_messages([
    ("system", "Ты — ассистент."),
    MessagesPlaceholder(variable_name="history", optional=True),
    ("human", "{input}")
])

# Можно вызывать без истории
result = prompt.invoke({"input": "Привет!"})
```

Если история не передана, placeholder просто пропускается.

### Управление длиной истории

Бесконечная история не помещается в контекстное окно модели. Нужны стратегии обрезки. LangChain предоставляет утилиты для этого:

```python
from langchain_core.messages import trim_messages

trimmed = trim_messages(
    history,
    max_tokens=1000,
    strategy="last",  # Оставить последние сообщения
    token_counter=model,  # Использовать модель для подсчёта токенов
    include_system=True  # Всегда сохранять системное сообщение
)
```

Функция `trim_messages` удаляет сообщения, чтобы уложиться в лимит токенов. Стратегия "last" оставляет последние сообщения (самые релевантные для текущего контекста). Стратегия "first" — первые (если важен начальный контекст).

Интеграция с LCEL:

```python
chain = (
    RunnablePassthrough.assign(
        history=lambda x: trim_messages(x["history"], max_tokens=1000, token_counter=model)
    )
    | prompt
    | model
)
```

Теперь история автоматически обрезается перед форматированием промпта.

## Часть 4: Partial formatting — частичное заполнение

### Зачем нужно частичное заполнение

Представьте промпт с несколькими переменными: `{system_context}`, `{user_name}`, `{question}`. Системный контекст и имя пользователя известны при инициализации приложения. Вопрос — только в момент запроса.

Можно передавать все переменные при каждом вызове, но это неудобно и подвержено ошибкам. Частичное заполнение позволяет "закрепить" известные значения заранее:

```python
base_prompt = ChatPromptTemplate.from_messages([
    ("system", "Контекст: {system_context}"),
    ("human", "Я — {user_name}. {question}")
])

# Частично заполняем известные значения
partial_prompt = base_prompt.partial(
    system_context="Приложение для поддержки клиентов",
    user_name="Алексей"
)

# Теперь нужно передать только question
result = partial_prompt.invoke({"question": "Как вернуть товар?"})
```

Partial возвращает новый промпт, где часть переменных уже заполнена. Оригинальный промпт не изменяется — это иммутабельная операция.

### Partial с функциями

Иногда значение не известно заранее, но вычисляется по формуле. Типичный пример — текущая дата и время:

```python
from datetime import datetime

prompt = ChatPromptTemplate.from_messages([
    ("system", "Сегодня {date}. Учитывай это при ответах."),
    ("human", "{question}")
])

def get_current_date():
    return datetime.now().strftime("%d %B %Y года")

partial_prompt = prompt.partial(date=get_current_date)
```

При каждом вызове `partial_prompt.invoke()` функция `get_current_date` выполняется заново, и текущая дата подставляется в промпт.

Это мощный паттерн для динамического контекста: текущее время, случайные числа, данные из внешних источников.

## Часть 5: Композиция промптов

### Объединение промптов

Промпты в LangChain можно объединять. Оператор `+` конкатенирует сообщения:

```python
system_prompt = ChatPromptTemplate.from_messages([
    ("system", "Ты — полезный ассистент.")
])

user_prompt = ChatPromptTemplate.from_messages([
    ("human", "{question}")
])

combined = system_prompt + user_prompt
# Эквивалентно:
# ChatPromptTemplate.from_messages([
#     ("system", "Ты — полезный ассистент."),
#     ("human", "{question}")
# ])
```

Это полезно для модульности. Можно хранить разные части промпта отдельно и комбинировать по необходимости.

### PipelinePromptTemplate

Для более сложных случаев существует PipelinePromptTemplate, позволяющий строить иерархии промптов:

```python
from langchain_core.prompts import PipelinePromptTemplate

# Базовые компоненты
persona = PromptTemplate.from_template("Ты — {role}.")
context = PromptTemplate.from_template("Контекст: {context_info}")
task = PromptTemplate.from_template("Задача: {task_description}")

# Финальный шаблон, собирающий компоненты
final = PromptTemplate.from_template("""
{persona}
{context}
{task}
Вопрос: {question}
""")

pipeline = PipelinePromptTemplate(
    final_prompt=final,
    pipeline_prompts=[
        ("persona", persona),
        ("context", context),
        ("task", task)
    ]
)

# Использование
result = pipeline.invoke({
    "role": "эксперт по Python",
    "context_info": "Пользователь — новичок",
    "task_description": "Объяснять простым языком",
    "question": "Что такое декоратор?"
})
```

PipelinePromptTemplate сначала форматирует каждый подпромпт, затем подставляет результаты в финальный шаблон. Это позволяет создавать сложные, переиспользуемые структуры промптов.

## Часть 6: Best practices и антипаттерны

### Структура эффективного промпта

Опытные prompt engineer'ы выработали набор практик, повышающих качество ответов модели.

**Чёткие инструкции в начале.** Системное сообщение должно ясно определять роль и ограничения модели. Модели лучше следуют инструкциям, расположенным в начале контекста.

```python
# Хорошо
system = """Ты — юридический консультант.
Твоя задача — отвечать на вопросы о трудовом законодательстве РФ.
Ограничения:
- Отвечай только на основе актуального законодательства
- Если вопрос вне твоей компетенции, скажи об этом
- Не давай финансовых советов"""

# Плохо
system = "Ты помощник"  # Слишком неопределённо
```

**Формат вывода.** Если нужен структурированный ответ, опишите формат явно:

```python
system = """Отвечай в формате JSON:
{
    "answer": "краткий ответ",
    "confidence": "число от 0 до 100",
    "sources": ["список источников"]
}"""
```

**Примеры желаемого поведения.** Few-shot примеры показывают модели ожидаемый паттерн лучше, чем абстрактные инструкции.

**Разделители контента.** Когда промпт содержит пользовательский ввод, отделяйте его визуально:

```python
prompt = """Проанализируй следующий текст:
---
{user_text}
---
Твой анализ:"""
```

Разделители (`---`, `###`, XML-теги) помогают модели понять структуру запроса.

### Антипаттерны

**Избыточная вежливость.** "Пожалуйста, не мог бы ты, если тебе не сложно, ответить на вопрос..." — это шум, который занимает токены и не улучшает качество.

**Противоречивые инструкции.** "Будь кратким, но давай подробные объяснения" — модель не знает, какую инструкцию приоритизировать.

**Негативные инструкции без альтернативы.** "Не используй технические термины" хуже, чем "Объясняй простым языком, доступным школьнику".

**Слишком длинные промпты.** Если промпт занимает половину контекстного окна, модели остаётся мало места для рассуждений и ответа. Это особенно критично для задач, требующих длинных ответов.

**Хардкод контекста.** Если контекст может меняться (дата, имя пользователя, язык), используйте переменные, а не зашивайте значения.

### Тестирование и итерация

Prompt engineering — итеративный процесс. Первый промпт редко бывает оптимальным. Рекомендации:

Создайте тестовый набор: коллекцию входов и ожидаемых выходов. Прогоняйте его при каждом изменении промпта.

Используйте версионирование. Храните промпты как файлы или в базе данных с версиями. Это позволяет откатиться, если новый промпт хуже.

Логируйте входы и выходы. Анализ реальных запросов пользователей — лучший источник идей для улучшения.

A/B тестирование. Для продакшн-систем тестируйте новые промпты на части трафика, сравнивайте метрики.

## Часть 7: Интеграция с LCEL

### Промпт как Runnable

ChatPromptTemplate реализует интерфейс Runnable. Это означает:

- Его можно использовать в pipe-выражениях
- Он поддерживает invoke, batch, stream
- Его можно комбинировать с другими Runnable

Типичный паттерн:

```python
chain = prompt | model | parser
```

Промпт принимает словарь с переменными, форматирует сообщения, передаёт модели. Модель генерирует ответ, парсер извлекает нужные данные.

### Входная схема промпта

Промпт знает, какие переменные ему нужны. Метод `input_schema` возвращает Pydantic-модель:

```python
prompt = ChatPromptTemplate.from_messages([
    ("system", "Ты эксперт в {domain}"),
    ("human", "{question}")
])

print(prompt.input_schema.schema())
# {'properties': {'domain': {'type': 'string'}, 'question': {'type': 'string'}},
#  'required': ['domain', 'question'], 'type': 'object'}
```

Это полезно для валидации и документации. Если вы забудете передать переменную, ошибка будет понятной.

### Композиция с RunnablePassthrough

Часто нужно обогатить входные данные перед передачей в промпт:

```python
chain = (
    RunnablePassthrough.assign(
        context=retriever,  # Добавляем результаты поиска
        timestamp=lambda _: datetime.now().isoformat()  # Добавляем время
    )
    | prompt
    | model
)
```

RunnablePassthrough.assign добавляет поля к входному словарю. Промпт получает и оригинальные данные, и обогащённые.

## Часть 8: Промпты для специальных задач

### Промпты для RAG

RAG (Retrieval-Augmented Generation) требует специфического формата промпта. Модель должна отвечать на основе предоставленного контекста, а не своих "знаний":

```python
rag_prompt = ChatPromptTemplate.from_messages([
    ("system", """Ты — ассистент, отвечающий на вопросы на основе предоставленного контекста.
Правила:
- Используй ТОЛЬКО информацию из контекста
- Если ответа нет в контексте, скажи "Я не нашёл информации об этом"
- Цитируй источники, указывая номер документа"""),
    ("human", """Контекст:
{context}

Вопрос: {question}""")
])
```

Ключевые элементы:

- Явное указание использовать только контекст
- Инструкция для случая, когда ответа нет
- Требование цитирования

### Промпты для агентов

Агентам нужны промпты, описывающие доступные инструменты и формат использования:

```python
agent_prompt = ChatPromptTemplate.from_messages([
    ("system", """Ты — AI-ассистент с доступом к инструментам.

Доступные инструменты:
{tools}

Чтобы использовать инструмент, верни JSON:
{{"tool": "название_инструмента", "input": {{"параметр": "значение"}}}}

Чтобы дать финальный ответ, верни JSON:
{{"final_answer": "твой ответ"}}"""),
    MessagesPlaceholder(variable_name="history"),
    ("human", "{input}"),
    MessagesPlaceholder(variable_name="agent_scratchpad")
])
```

`agent_scratchpad` — специальный placeholder для промежуточных рассуждений агента (его предыдущие действия и наблюдения).

### Промпты для классификации

Классификация требует чёткого определения категорий:

```python
classification_prompt = ChatPromptTemplate.from_messages([
    ("system", """Классифицируй обращение клиента.

Категории:
- BILLING: вопросы об оплате, счетах, тарифах
- TECHNICAL: технические проблемы, баги, настройка
- SALES: интерес к покупке, апгрейд, скидки
- COMPLAINT: жалобы, недовольство
- OTHER: не относится к другим категориям

Верни ТОЛЬКО название категории, одно слово."""),
    ("human", "{message}")
])
```

Чёткое определение категорий минимизирует неопределённость и повышает консистентность классификации.

## Заключение: Промпт как артефакт

Промпт — это не строка текста, которую пишут один раз и забывают. Это артефакт разработки, требующий такого же внимания, как код. Хороший промпт должен быть:

- **Чётким**: Ясные инструкции, однозначный формат
- **Модульным**: Переиспользуемые компоненты, параметризация
- **Тестируемым**: Набор тестовых кейсов, метрики качества
- **Версионируемым**: История изменений, возможность отката
- **Документированным**: Объяснение логики, примеры использования

LangChain предоставляет инструменты для всего этого: ChatPromptTemplate для структуры, MessagesPlaceholder для динамики, partial для параметризации, интеграция с LCEL для композиции.

## Ключевые термины

| Термин | Определение |
|--------|-------------|
| **ChatPromptTemplate** | Основной класс для создания шаблонов промптов с сообщениями разных ролей |
| **MessagesPlaceholder** | Placeholder для вставки динамического списка сообщений (например, истории) |
| **SystemMessage** | Сообщение с ролью System — инструкции для модели |
| **HumanMessage** | Сообщение с ролью Human — ввод пользователя |
| **AIMessage** | Сообщение с ролью AI — предыдущие ответы модели |
| **Few-shot prompting** | Техника включения примеров желаемого поведения в промпт |
| **Partial formatting** | Частичное заполнение шаблона с фиксацией известных переменных |
| **ExampleSelector** | Механизм динамического выбора релевантных примеров для few-shot |
