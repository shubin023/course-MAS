# Раздел 3: Векторные базы данных — Хранилища смыслов

## Введение: Когда обычные базы данных не справляются

Представьте себе библиотеку с миллионом книг. В традиционной библиотеке книги организованы по системе: алфавит авторов, тематические каталоги, ISBN. Если вы знаете автора или точное название — найдёте быстро. Но что если ваш запрос звучит так: "Хочу что-то похожее на 'Мастера и Маргариту', но про современность"? Традиционный каталог бессилен — он не понимает "похожести".

Именно такую задачу решают векторные базы данных. Они хранят не просто данные, а их семантические "тени" — векторные представления. И умеют искать не по точному совпадению, а по близости в пространстве смыслов. Запрос "документы про удалённую работу" найдёт текст с заголовком "Регламент дистанционного формата", даже если слова "удалённая" и "работа" там не встречаются.

В предыдущем разделе научились превращать тексты в векторы. Теперь нужно понять, как хранить миллионы этих векторов и искать среди них за миллисекунды. Задача нетривиальная: наивный перебор всех векторов для вычисления сходства при миллионе документов займёт секунды — неприемлемо для интерактивных приложений.

## Часть 1: Проблема поиска ближайших соседей

### Наивный подход: полный перебор

Самый простой способ найти ближайшие к запросу векторы — сравнить запрос со всеми векторами в базе. Это называется brute-force или exact search.

```python
import numpy as np

def brute_force_search(query_vector, all_vectors, k=5):
    # Вычисляем сходство со всеми векторами
    similarities = np.dot(all_vectors, query_vector)
    # Находим индексы k максимальных значений
    top_k_indices = np.argsort(similarities)[-k:][::-1]
    return top_k_indices
```

Сложность: O(n × d), где n — количество векторов, d — размерность. Для миллиона 1024-мерных векторов это миллиард операций на запрос.

На современном CPU с SIMD-оптимизациями это примерно 100-500 мс. Кажется терпимо? Но представьте 100 запросов в секунду — сервер захлебнётся. А если векторов 100 миллионов?

### Approximate Nearest Neighbors (ANN)

Ключевая идея: мы готовы пожертвовать гарантией нахождения идеальных ближайших соседей ради скорости. Если из 5 "настоящих" ближайших мы найдём 4 из 5 — это часто достаточно хорошо для RAG.

ANN-алгоритмы строят специальные индексные структуры, позволяющие отсечь большинство кандидатов без вычисления сходства. Вместо сравнения с миллионом векторов мы сравниваем с тысячей — ускорение в 1000 раз!

Качество ANN измеряется метрикой recall@k: какая доля "истинных" k ближайших соседей найдена алгоритмом. Recall 0.95 означает, что в среднем 95% настоящих ближайших соседей попадают в результат.

Компромисс "скорость vs качество" — центральная тема проектирования векторных баз данных.

## Часть 2: Алгоритмы приближённого поиска

### IVF: Инвертированный файловый индекс

IVF (Inverted File Index) — один из старейших и простейших ANN-алгоритмов. Идея: разбить всё пространство на кластеры, при поиске проверять только несколько ближайших кластеров.

**Построение индекса:**
1. Выбираем количество кластеров (центроидов), например 1000
2. Запускаем k-means кластеризацию на всех векторах
3. Каждый вектор приписывается к своему кластеру

**Поиск:**
1. Находим ближайшие к запросу центроиды (например, 10 из 1000)
2. Перебираем только векторы внутри этих кластеров
3. Вычисляем точное сходство для отобранных кандидатов

Если в каждом кластере ~1000 векторов, и мы проверяем 10 кластеров, вместо миллиона сравнений делаем 10 000 — ускорение в 100 раз.

Параметр nprobe контролирует количество проверяемых кластеров: больше nprobe — лучше recall, но медленнее.

**Визуализация:** Представьте карту города, разбитую на районы. Вместо обхода всех домов вы сначала определяете интересующие районы (по близости к точке), затем ищете только внутри них.

### HNSW: Иерархические навигационные сети малого мира

HNSW (Hierarchical Navigable Small World) — современный стандарт для ANN. Это граф, в котором узлы — векторы, а рёбра соединяют "соседей".

**Концепция малого мира:** В социальных сетях любых двух людей на Земле разделяет в среднем 6 рукопожатий ("шесть степеней разделения"). Это свойство "малого мира" — даже в огромном графе можно быстро добраться от любого узла к любому.

HNSW строит многоуровневый граф:
- Верхние уровни — разреженные, с "дальними" связями для быстрой навигации
- Нижние уровни — плотные, с "близкими" связями для точного поиска

**Поиск:** Начинаем с верхнего уровня, жадно переходим к ближайшему к запросу соседу, спускаемся на уровень ниже, повторяем. На нижнем уровне делаем более тщательный локальный поиск.

HNSW обычно даёт лучший компромисс качества и скорости среди всех ANN-алгоритмов. Главный минус — потребление памяти (графовая структура требует хранения связей).

**Параметры:**
- M: количество связей на узел (больше — лучше recall, больше памяти)
- efConstruction: качество построения индекса (больше — лучше индекс, дольше строится)
- efSearch: качество поиска (больше — лучше recall, медленнее)

### Сравнение IVF и HNSW

| Аспект | IVF | HNSW |
|--------|-----|------|
| **Скорость поиска** | Хорошая | Отличная |
| **Recall** | Средний | Высокий |
| **Память** | Низкая | Высокая |
| **Время построения** | Быстро | Медленно |
| **Обновляемость** | Пересборка кластеров | Инкрементальная |
| **Когда использовать** | Ограниченная память, статичные данные | Высокие требования к качеству |

### Product Quantization (PQ): сжатие для масштаба

Что если памяти не хватает даже для хранения самих векторов? Product Quantization сжимает векторы, разбивая их на сегменты и квантуя каждый сегмент отдельно.

**Идея:**
1. Вектор 1024 измерений разбивается на 8 сегментов по 128 измерений
2. Для каждого сегмента обучается кодовая книга из 256 "типичных" подвекторов
3. Каждый сегмент заменяется индексом ближайшего элемента кодовой книги

Результат: вместо 1024 × 4 = 4096 байт на вектор храним 8 × 1 = 8 байт — сжатие в 512 раз!

Расстояние между сжатыми векторами приблизительное, но для ANN этого достаточно. PQ часто комбинируют с IVF: IVF сужает кандидатов, PQ позволяет хранить их компактно.

## Часть 3: Обзор векторных баз данных

### ChromaDB: простота для старта

ChromaDB — самый простой способ начать работу с векторным поиском. Создан специально для RAG и AI-приложений.

**Характеристики:**
- Встраиваемая (embedded) БД — работает в процессе Python, не требует отдельного сервера
- С версии 1.5.0 поддерживает полноценный серверный режим (`chroma run --path /db_path`)
- Отдельный HTTP-клиент (`chromadb-client`) для client-server архитектуры
- Минималистичный API
- Поддержка метаданных и фильтрации
- Сохранение на диск (persistence)
- Улучшения производительности до 70% в последних версиях

**Ограничения embedded режима:**
- Не масштабируется горизонтально (в embedded режиме)
- Ограниченная производительность (до миллионов векторов)

**Когда использовать:** прототипы, локальная разработка, небольшие проекты. Для масштабирования используйте server mode.

```python
import chromadb
from chromadb.utils import embedding_functions

# Создание клиента
client = chromadb.PersistentClient(path="./chroma_db")

# Определение функции эмбеддинга
ef = embedding_functions.SentenceTransformerEmbeddingFunction(
    model_name="BAAI/bge-m3"
)

# Создание или получение коллекции
collection = client.get_or_create_collection(
    name="documents",
    embedding_function=ef
)

# Добавление документов
collection.add(
    documents=["Политика удалённой работы...", "Процедура оформления отпуска..."],
    metadatas=[{"category": "HR"}, {"category": "HR"}],
    ids=["doc1", "doc2"]
)

# Поиск
results = collection.query(
    query_texts=["Как оформить отпуск?"],
    n_results=5,
    where={"category": "HR"}  # Фильтрация по метаданным
)
```

### FAISS: мощность от Meta

FAISS (Facebook AI Similarity Search) — библиотека от Meta, ставшая индустриальным стандартом для ANN.

**Характеристики:**
- Высочайшая производительность (оптимизация под CPU и GPU)
- Множество алгоритмов: IVF, HNSW, PQ, их комбинации
- Поддержка GPU для ускорения
- Обрабатывает миллиарды векторов

**Ограничения:**
- Низкоуровневый API, не база данных в полном смысле
- Нет встроенной persistence (нужно вручную сохранять/загружать)
- Нет метаданных и фильтрации "из коробки"
- Однопоточный без дополнительных обёрток

**Когда использовать:** когда нужна максимальная производительность, готовы инвестировать в инфраструктуру.

```python
import faiss
import numpy as np

# Параметры
dimension = 768
n_vectors = 1_000_000

# Создание данных
vectors = np.random.random((n_vectors, dimension)).astype('float32')
faiss.normalize_L2(vectors)  # Нормализация для косинусного сходства

# Создание индекса HNSW
index = faiss.IndexHNSWFlat(dimension, 32)  # 32 = M parameter
index.hnsw.efConstruction = 200  # Качество построения

# Добавление векторов
index.add(vectors)

# Поиск
query = np.random.random((1, dimension)).astype('float32')
faiss.normalize_L2(query)

index.hnsw.efSearch = 100  # Качество поиска
distances, indices = index.search(query, k=10)
```

### Pinecone: полностью управляемый сервис

Pinecone — cloud-native векторная база данных как сервис.

**Характеристики:**
- Полностью управляемый (serverless или managed pods)
- Автоматическое масштабирование
- Встроенные метаданные и фильтрация
- Гибридный поиск (vectors + keywords)
- Удобный API и интеграции

**Ограничения:**
- Облачный — данные уходят в Pinecone
- Платный (есть бесплатный tier с ограничениями)
- Vendor lock-in

**Когда использовать:** когда важна простота эксплуатации, есть бюджет, нет жёстких требований к приватности.

```python
from pinecone import Pinecone

# Инициализация
pc = Pinecone(api_key="your-api-key")
index = pc.Index("my-index")

# Добавление векторов
index.upsert(
    vectors=[
        {"id": "doc1", "values": [...], "metadata": {"category": "HR"}},
        {"id": "doc2", "values": [...], "metadata": {"category": "Finance"}}
    ]
)

# Поиск с фильтрацией
results = index.query(
    vector=[...],
    top_k=5,
    filter={"category": {"$eq": "HR"}},
    include_metadata=True
)
```

**Примечание о версиях:** Pinecone Python SDK v8 (февраль 2026) требует Python 3.10 или выше. Python 3.9 больше не поддерживается.

### Qdrant: золотая середина

Qdrant — open-source векторная БД с акцентом на production-ready функции.

**Характеристики:**
- Open-source с enterprise-версией
- Rust под капотом (производительность)
- Богатая фильтрация по метаданным
- Полноценный REST и gRPC API
- Docker и Kubernetes deployment
- Гибридный поиск

**Ограничения:**
- Сложнее в настройке, чем ChromaDB
- Меньше экосистема, чем у Pinecone

**Когда использовать:** production с требованиями к приватности, средние и большие объёмы данных.

```python
from qdrant_client import QdrantClient
from qdrant_client.models import VectorParams, Distance, PointStruct

# Подключение
client = QdrantClient(host="localhost", port=6333)

# Создание коллекции
client.create_collection(
    collection_name="documents",
    vectors_config=VectorParams(
        size=768,
        distance=Distance.COSINE
    )
)

# Добавление точек
client.upsert(
    collection_name="documents",
    points=[
        PointStruct(
            id=1,
            vector=[...],
            payload={"text": "...", "category": "HR"}
        )
    ]
)

# Поиск с фильтрацией
results = client.search(
    collection_name="documents",
    query_vector=[...],
    limit=5,
    query_filter={
        "must": [{"key": "category", "match": {"value": "HR"}}]
    }
)
```

## Часть 4: Метаданные и фильтрация

### Зачем нужны метаданные

Чистый векторный поиск находит семантически похожие документы, но иногда этого недостаточно. Примеры:

- "Найди документы про отпуск, но только из HR-раздела"
- "Статьи про машинное обучение за последний год"
- "Письма от конкретного отправителя на заданную тему"

Метаданные — структурированные атрибуты документов — позволяют комбинировать семантический поиск с фильтрацией.

### Типы метаданных

**Категориальные:** отдел, тип документа, автор, тег. Фильтрация по точному совпадению или вхождению в список.

**Числовые:** дата (timestamp), версия, рейтинг, размер файла. Фильтрация по диапазону.

**Булевы:** is_public, is_archived, has_attachments.

**Текстовые:** заголовок, резюме. Могут использоваться для keyword-поиска в гибридных системах.

### Стратегии фильтрации

**Pre-filtering:** сначала фильтруем по метаданным, потом ищем по вектору среди отфильтрованных. Быстро, если фильтр сильно сужает выборку. Проблема: если после фильтра осталось мало документов, recall страдает.

**Post-filtering:** сначала ищем топ-N по вектору, потом фильтруем. Риск: если релевантные документы не проходят фильтр, результат пуст.

**Integrated filtering:** фильтрация встроена в алгоритм ANN. Современные БД (Pinecone, Qdrant, Weaviate) поддерживают это — наиболее эффективный подход.

```python
# Пример интегрированной фильтрации в Qdrant
results = client.search(
    collection_name="documents",
    query_vector=query_embedding,
    limit=10,
    query_filter={
        "must": [
            {"key": "department", "match": {"value": "HR"}},
            {"key": "year", "range": {"gte": 2023}}
        ],
        "must_not": [
            {"key": "status", "match": {"value": "archived"}}
        ]
    }
)
```

## Часть 5: Индексирование и обновление

### Построение индекса

Индексирование (создание структур для быстрого поиска) — дорогая операция. Для миллиона векторов HNSW-индекс строится от минут до часов в зависимости от параметров.

**Batch vs Streaming:**
- Batch: добавляем все документы сразу, строим индекс один раз. Эффективнее для начальной загрузки.
- Streaming: добавляем документы по одному. HNSW поддерживает инкрементальное добавление, IVF требует периодической пересборки.

### Обновление данных

Что если документ изменился?

**Immutable подход:** удаляем старый вектор, добавляем новый. Просто концептуально, но удаление в некоторых индексах (HNSW) дорогое.

**Versioning:** храним все версии, фильтруем по "текущей" версии. Больше памяти, но проще управлять.

**Soft delete:** помечаем документ как удалённый, периодически перестраиваем индекс без помеченных.

### Multi-tenancy: изоляция данных

В многопользовательских приложениях нужно изолировать данные разных клиентов.

**Отдельные коллекции:** каждый tenant — отдельная коллекция/индекс. Простая изоляция, но overhead на управление множеством индексов.

**Фильтрация по tenant_id:** все данные в одном индексе, tenant_id в метаданных. Эффективнее для большого числа мелких tenants, но нужна аккуратная настройка фильтрации.

**Hybrid:** крупные tenants — отдельные коллекции, мелкие — объединены.

## Часть 6: Гибридный поиск

### Зачем совмещать vector и keyword

Векторный поиск хорош для семантики: "документы про увольнение" найдёт "процедура расторжения трудового договора". Но плох для точных терминов: запрос "ISO 27001" должен находить документы с этим точным обозначением, а не семантически похожие (про безопасность в целом).

Keyword-поиск (BM25, TF-IDF) наоборот: отлично находит точные совпадения, плохо справляется с перефразированием.

Гибридный поиск комбинирует оба подхода: используем и векторное сходство, и текстовый поиск, объединяем результаты.

### Reciprocal Rank Fusion (RRF)

Как объединить два ранжированных списка? RRF — простой и эффективный метод.

Для каждого документа вычисляем RRF-скор:

$$RRF(d) = \sum_{r \in R} \frac{1}{k + rank_r(d)}$$

где R — множество ранжирований (vector search, BM25), rank_r(d) — позиция документа d в ранжировании r, k — сглаживающий параметр (обычно 60).

Документы, появляющиеся высоко в обоих списках, получают высокий RRF-скор. Документы высоко в одном, но низко в другом — средний. Документы, отсутствующие в каком-то списке — штрафуются.

```python
def reciprocal_rank_fusion(rankings, k=60):
    """
    rankings: list of lists, каждый список — ранжирование (doc_id по убыванию релевантности)
    returns: sorted list of (doc_id, rrf_score)
    """
    rrf_scores = {}

    for ranking in rankings:
        for rank, doc_id in enumerate(ranking, start=1):
            if doc_id not in rrf_scores:
                rrf_scores[doc_id] = 0
            rrf_scores[doc_id] += 1 / (k + rank)

    # Сортировка по убыванию скора
    return sorted(rrf_scores.items(), key=lambda x: x[1], reverse=True)

# Пример
vector_results = ["doc3", "doc1", "doc5", "doc2"]
bm25_results = ["doc1", "doc4", "doc3", "doc6"]

combined = reciprocal_rank_fusion([vector_results, bm25_results])
# doc1 высоко в обоих → топ результат
# doc3 высоко в vector, средне в bm25 → второй-третий
# doc4 только в bm25 → ниже
```

### Реализация в векторных БД

**Weaviate:** встроенный гибридный поиск с настраиваемым alpha (баланс vector/BM25).

```python
# Weaviate hybrid search
response = client.query.get("Document", ["content"]).with_hybrid(
    query="ISO 27001 безопасность данных",
    alpha=0.5  # 0 = только BM25, 1 = только vector
).with_limit(10).do()
```

**Qdrant:** отдельные sparse vectors для BM25, комбинация через RRF.

**Pinecone:** sparse-dense vectors, встроенный hybrid.

## Часть 7: Практические рекомендации

### Выбор базы данных: чек-лист

1. **Объём данных:**
   - `<100K документов: ChromaDB достаточно`
   - `100K-10M: Qdrant, Weaviate, Pinecone`
   - `>10M: FAISS, Milvus, или managed Pinecone`

2. **Требования к приватности:**
   - Данные не могут покидать инфраструктуру → self-hosted (Qdrant, Weaviate, Milvus)
   - Приватность не критична → managed (Pinecone) проще

3. **Бюджет на DevOps:**
   - Минимальный → ChromaDB или managed Pinecone
   - Есть команда → self-hosted даёт больше контроля

4. **Сложность запросов:**
   - Только vector search → любая
   - Сложная фильтрация → Qdrant, Weaviate (мощная query language)
   - Гибридный поиск → Weaviate, Pinecone

### Оптимизация производительности

**Нормализуйте векторы заранее.** Если используете cosine similarity, нормализуйте при индексации — тогда можно использовать dot product (быстрее).

**Настройте параметры индекса под ваш recall/latency tradeoff.** Начните с дефолтов, затем экспериментируйте с efSearch/nprobe.

**Используйте батчинг.** Добавляйте документы пачками по 100-1000, не по одному.

**Мониторьте.** Следите за latency p50/p95/p99, recall на тестовых запросах.

### Типичные ошибки

**Ошибка 1: Игнорирование размерности.** 4096-мерные эмбеддинги красивы, но занимают много памяти и медленнее в поиске. Часто 768 или даже 384 измерений достаточно.

**Ошибка 2: Слишком много метаданных.** Каждое поле метаданных — дополнительный индекс. Храните только то, по чему реально фильтруете.

**Ошибка 3: Редкая переиндексация.** Если используете IVF и данные меняются — кластеры устаревают. Планируйте периодическую пересборку.

**Ошибка 4: Недостаточное тестирование recall.** "Работает быстро" — недостаточно. Нужно измерять, находит ли система релевантные документы.

## Часть 8: Интеграция с LangChain

### LangChain VectorStore абстракция

LangChain предоставляет унифицированный интерфейс для работы с разными векторными БД.

```python

from langchain_community.vectorstores import Chroma, Qdrant, Pinecone, FAISS
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings()

# Все используют одинаковый интерфейс:
vectorstore = Chroma.from_documents(docs, embeddings)
# или
vectorstore = Qdrant.from_documents(docs, embeddings, url="http://localhost:6333")
# или
vectorstore = FAISS.from_documents(docs, embeddings)

# Поиск — одинаковый метод
results = vectorstore.similarity_search("Как оформить отпуск?", k=5)

# С фильтрацией (где поддерживается)
results = vectorstore.similarity_search(
    "Как оформить отпуск?",
    k=5,
    filter={"department": "HR"}
)

```

### Retriever API

Для интеграции с RAG-цепочками используется Retriever:

```python
# Базовый retriever
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})

# С фильтром
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={
        "k": 10,
        "filter": {"category": "policy"}
    }
)

# С порогом сходства
retriever = vectorstore.as_retriever(
    search_type="similarity_score_threshold",
    search_kwargs={
        "score_threshold": 0.7,
        "k": 10
    }
)

# Использование в цепочке
from langchain_core.runnables import RunnablePassthrough

chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | prompt
    | llm
)
```

## Заключение: Векторные БД как инфраструктурный слой RAG

Пройден путь от понимания проблемы (поиск среди миллионов векторов) через алгоритмы (IVF, HNSW, PQ) к конкретным решениям (ChromaDB, FAISS, Pinecone, Qdrant). Векторные базы данных — это инфраструктурный фундамент, на котором строится retrieval.

Ключевые идеи:

**ANN — компромисс точности и скорости.** Мы согласны не найти идеальных ближайших соседей, если найдём почти идеальных в 100 раз быстрее.

**HNSW — текущий стандарт** для большинства задач, балансирующий качество, скорость и инкрементальность.

**Выбор БД зависит от контекста:** объём данных, требования к приватности, бюджет на инфраструктуру, сложность запросов.

**Метаданные и гибридный поиск** расширяют возможности: структурированная фильтрация + семантика + ключевые слова.

## Ключевые термины

| Термин | Определение |
|--------|-------------|
| **ANN** | Approximate Nearest Neighbors — алгоритмы приближённого поиска ближайших соседей |
| **IVF** | Inverted File Index — ANN-алгоритм на основе кластеризации |
| **HNSW** | Hierarchical Navigable Small World — граф-based ANN-алгоритм |
| **Product Quantization** | Метод сжатия векторов через квантизацию сегментов |
| **Recall@k** | Доля истинных ближайших соседей, найденных в топ-k результатов |
| **Метаданные** | Структурированные атрибуты документов для фильтрации |
| **Гибридный поиск** | Комбинация векторного и keyword-based поиска |
| **RRF** | Reciprocal Rank Fusion — метод объединения ранжированных списков |
