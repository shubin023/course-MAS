# Раздел 1.2: Анатомия AI-агента

## Введение: Метафора живого организма

Когда мы говорим об «анатомии» агента, мы не просто используем красивую метафору. Как живой организм состоит из взаимосвязанных систем — нервной, кровеносной, мышечной, — так и современный AI-агент представляет собой композицию специализированных компонентов, каждый из которых выполняет свою функцию, но не может существовать изолированно от других. Нервная система бесполезна без мышц, которые приводят тело в движение; мозг бесполезен без органов чувств, которые доставляют ему информацию о мире. Точно так же языковая модель — сколь бы мощной она ни была — остаётся лишь «мозгом в банке», пока у неё нет инструментов для взаимодействия с миром и памяти для сохранения контекста.

---

## Определение агентности: три ключевых свойства

Прежде чем погружаться в компоненты, определимся с тем, что отличает агента от других программных систем. Агентность — это не бинарное свойство («агент или не агент»), а скорее спектр, и разные системы обладают ею в разной степени.

Первое свойство агента — автономность. Агент способен действовать без постоянного руководства со стороны человека. Это не означает, что агент полностью независим — большинство практических агентов работают в режиме «human-in-the-loop», запрашивая подтверждение на критических этапах. Но агент сам решает, когда и какие действия предпринять, а не просто выполняет заранее запрограммированную последовательность шагов.

Второе свойство — целенаправленность. Агент преследует цель и способен оценивать, приближается он к ней или удаляется. Это принципиально отличает агента от реактивной системы, которая просто отвечает на входные данные. Чат-бот, отвечающий на вопросы, — реактивная система. Агент, который ищет информацию в интернете, анализирует её и составляет отчёт, — целенаправленная система.

Третье свойство — способность к восприятию и воздействию. Агент взаимодействует со средой: получает от неё информацию (через «сенсоры») и влияет на неё (через «актуаторы»). В случае AI-агентов «сенсорами» могут быть API для получения данных, инструменты для чтения файлов или веб-страниц, а «актуаторами» — API для отправки сообщений, создания файлов, выполнения кода.

Эти три свойства можно формализовать через цикл взаимодействия агента со средой. На каждом шаге агент находится в некотором состоянии, воспринимает информацию о среде, принимает решение о действии и выполняет его, после чего переходит в новое состояние. Этот цикл продолжается до достижения цели (или до превышения лимита шагов, или до ошибки — мы обсудим это подробнее).

## Цикл OODA: военная стратегия на службе AI

Для понимания того, как агент принимает решения, полезно обратиться к модели, разработанной далеко от мира искусственного интеллекта. В 1970-х годах американский военный стратег Джон Бойд создал концепцию цикла OODA для описания процесса принятия решений в воздушном бою. Аббревиатура расшифровывается как Observe (Наблюдай), Orient (Ориентируйся), Decide (Решай), Act (Действуй).

Этап наблюдения (Observe) — это сбор информации о текущем состоянии среды. Для пилота истребителя это визуальное наблюдение за противником, показания приборов, радиоперехват. Для AI-агента — получение входных данных от пользователя, результаты вызова инструментов, содержимое памяти.

Этап ориентации (Orient) — самый сложный и важный. Здесь агент интерпретирует собранную информацию в контексте своих знаний, целей и ограничений. Бойд подчёркивал, что ориентация — это не просто анализ данных, а синтез, учитывающий культурные традиции, предыдущий опыт, генетическое наследие (в случае человека) или обучающие данные (в случае ИИ). Для AI-агента этап ориентации — это работа языковой модели, которая «понимает» ситуацию и формирует её внутреннее представление.

Этап решения (Decide) — выбор конкретного действия из множества возможных. Пилот решает, атаковать ли противника, уклониться, запросить поддержку. Агент решает, какой инструмент вызвать, какой запрос сформировать, достаточно ли собранной информации для ответа пользователю.

Этап действия (Act) — выполнение выбранного решения. Пилот нажимает на гашетку или тянет штурвал. Агент вызывает API, генерирует текст, сохраняет информацию в память.

Ключевая идея Бойда состояла в том, что в конкурентной среде побеждает тот, кто проходит цикл OODA быстрее противника. Если вы успеваете наблюдать, ориентироваться, решать и действовать до того, как противник завершит свой цикл, вы получаете преимущество. Ваши действия меняют ситуацию, и противнику приходится начинать свой цикл заново с устаревшими данными.

В контексте AI-агентов эта идея трансформируется: важна не столько скорость одного цикла, сколько качество ориентации и способность к многократному итерированию. Агент, который глубоко «понимает» задачу (качественная ориентация) и может многократно уточнять своё понимание через взаимодействие со средой (итерации цикла), превосходит агента, который действует быстро, но поверхностно.

## Четыре компонента агента: LLM, Tools, Memory, Planning

Теперь перейдём к конкретной архитектуре. Современный AI-агент состоит из четырёх основных компонентов, которые можно уподобить системам человеческого тела.

**LLM (Large Language Model) — мозг агента.** Языковая модель выполняет центральную функцию: интерпретирует входные данные, рассуждает о задаче, принимает решения о действиях и генерирует выходные данные. Это единственный компонент, который невозможно заменить чем-то принципиально иным — без LLM нет агента в современном понимании. Языковая модель определяет «интеллект» агента: его способность понимать нюансы, справляться с неоднозначностью, адаптироваться к новым ситуациям.

**Tools (инструменты) — руки агента.** Инструменты позволяют агенту воздействовать на внешний мир и получать от него информацию. Без инструментов агент остаётся чистой мыслью, неспособной ничего изменить. Инструменты могут быть простыми (калькулятор, генератор случайных чисел) или сложными (полный доступ к файловой системе, возможность выполнять произвольный код), безопасными (чтение публичной информации) или рискованными (отправка электронной почты, финансовые транзакции). Набор доступных инструментов определяет «возможности» агента — то, что он может делать.

**Memory (память) — хранилище опыта.** Память позволяет агенту сохранять информацию между шагами рассуждения и между сессиями взаимодействия. Без памяти каждый шаг начинается с чистого листа, и агент не может учиться на своих ошибках или учитывать контекст предыдущих взаимодействий. Память бывает краткосрочной (в пределах одного разговора) и долгосрочной (между сессиями), структурированной (базы данных, графы знаний) и неструктурированной (текстовые логи, векторные хранилища).

**Planning (планирование) — стратегическое мышление.** Планирование позволяет агенту декомпозировать сложные задачи на подзадачи, определять порядок их выполнения и адаптировать план по мере получения новой информации. Без планирования агент действует «близоруко», оптимизируя каждый шаг локально, но не видя общей картины. Планирование особенно критично для задач, где ошибка на раннем этапе делает невозможным успех на поздних этапах.

Эти четыре компонента не изолированы друг от друга — они постоянно взаимодействуют. LLM использует память для контекста, обращается к инструментам для действий, следует плану для структурирования работы. План может корректироваться на основе результатов использования инструментов. Память пополняется на каждом шаге. Это динамическая система с множеством обратных связей.

## LLM: мозг агента в деталях

Языковая модель — это сердце агента, и её характеристики напрямую определяют качество работы всей системы. Давайте разберём, какие свойства LLM критичны для агентных приложений.

**Способность следовать инструкциям (instruction following).** Агент получает задачи в виде текстовых инструкций, и модель должна точно понимать, что от неё требуется. Это может показаться тривиальным, но на практике модели часто «интерпретируют» инструкции по-своему, добавляя или убирая требования. Качественная модель для агентов должна уметь следовать сложным, многошаговым, условным инструкциям буквально.

**Способность к структурированному выводу (structured output).** Агент часто должен генерировать не произвольный текст, а данные в определённом формате: JSON для вызова инструментов, конкретные теги для разметки рассуждений, код на определённом языке. Модель должна надёжно соблюдать требуемый формат, не «сбиваясь» на свободный текст.

**Способность к рассуждению (reasoning).** Для решения сложных задач модель должна уметь «думать» — разбивать проблему на части, выявлять противоречия, делать выводы из фактов. Эта способность особенно важна на этапе ориентации в цикле OODA.

**Знание о мире (world knowledge).** Модель должна обладать достаточными знаниями о предметной области, в которой работает агент. Агент для юридического анализа требует модели, знакомой с правовыми концепциями; агент для написания кода — модели, обученной на качественном коде.

**Понимание контекста (context understanding).** Агентные сценарии часто требуют длинного контекста: история разговора, результаты предыдущих действий, содержимое анализируемых документов. Модель должна эффективно использовать весь доступный контекст, не «теряя» важную информацию.

На практике выбор модели — это всегда компромисс. Более мощные модели (GPT-4, Claude 3 Opus) обеспечивают лучшее качество рассуждений, но стоят дороже и работают медленнее. Менее мощные модели (GPT-3.5, Claude 3 Haiku) быстрее и дешевле, но чаще ошибаются в сложных случаях. Многие production-системы используют каскадирование: простые запросы обрабатывает дешёвая модель, а сложные перенаправляются к более мощной.

## Tools: расширение возможностей агента

Инструменты — это интерфейс между агентом и внешним миром. Они превращают языковую модель из «оракула», выдающего текст, в «оператора», способного действовать. Давайте рассмотрим, как устроены инструменты с технической точки зрения.

Каждый инструмент описывается тремя компонентами. Первый — имя, уникальный идентификатор инструмента, по которому агент на него ссылается. Второй — описание, текст на естественном языке, объясняющий, что инструмент делает и когда его следует использовать. Третий — схема параметров, формальное описание входных данных, которые инструмент принимает (обычно в формате JSON Schema).

Вот пример описания простого инструмента для получения погоды:

```json
{
  "name": "get_weather",
  "description": "Получает текущую погоду для указанного города. Используй этот инструмент, когда пользователь спрашивает о погоде.",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "Название города, например 'Москва' или 'New York'"
      },
      "units": {
        "type": "string",
        "enum": ["celsius", "fahrenheit"],
        "description": "Единицы измерения температуры"
      }
    },
    "required": ["city"]
  }
}
```

Когда агент решает использовать инструмент, он генерирует вызов в формате JSON:

```json
{
  "name": "get_weather",
  "arguments": {
    "city": "Москва",
    "units": "celsius"
  }
}
```

Оркестрирующий код парсит этот JSON, вызывает соответствующую функцию с указанными аргументами и возвращает результат агенту. Результат обычно тоже представляется в текстовом или JSON-формате.

Инструменты можно классифицировать по нескольким измерениям. По направлению воздействия: инструменты чтения (получение информации из среды) и инструменты записи (изменение среды). По уровню риска: безопасные инструменты (не имеют побочных эффектов или эффекты обратимы) и рискованные (необратимые действия, финансовые последствия). По области применения: универсальные (поиск в интернете, работа с файлами) и специализированные (API конкретного сервиса).

Хорошо спроектированный набор инструментов обладает несколькими свойствами. Полнота означает, что инструментов достаточно для решения целевых задач. Минимальность — нет избыточных инструментов с пересекающейся функциональностью, что может запутать агента. Ясность — описания инструментов однозначны, и агент понимает, когда какой инструмент использовать. Безопасность — рискованные операции требуют явного подтверждения или недоступны вовсе.

## Memory: преодоление беспамятства

Одно из фундаментальных ограничений языковых моделей — отсутствие встроенной памяти. Каждый вызов модели начинается «с чистого листа»: модель не помнит, о чём вы говорили минуту назад, если эта информация не включена в текущий промпт. Для агентов это создаёт серьёзную проблему: как сохранять контекст на протяжении длинного взаимодействия?

Решение — явное управление памятью на уровне приложения. Память агента обычно делится на несколько типов.

**Рабочая память (working memory)** — информация, активно используемая в текущем рассуждении. Это аналог оперативной памяти компьютера или содержимого сознания человека. В контексте LLM-агентов рабочая память — это, по сути, контент, который передаётся модели в каждом запросе: текущий промпт, история сообщений, результаты недавних действий.

**Эпизодическая память (episodic memory)** — записи о прошлых событиях и взаимодействиях. Это история разговоров, логи выполненных действий, кэш результатов поиска. Эпизодическая память позволяет агенту «помнить», что произошло ранее в текущей сессии или в прошлых сессиях.

**Семантическая память (semantic memory)** — общие знания о мире и предметной области. Это могут быть базы знаний, графы сущностей, индексы документов. Семантическая память позволяет агенту обращаться к информации, которая не входит в его обучающие данные или слишком объёмна для включения в промпт.

**Процедурная память (procedural memory)** — знание о том, как выполнять определённые действия. В контексте LLM-агентов это могут быть шаблоны промптов, описания инструментов, примеры успешных решений (few-shot examples).

Практическая реализация памяти сталкивается с фундаментальным ограничением: контекстное окно модели имеет конечный размер. Нельзя передать модели всю историю всех взаимодействий. Поэтому критически важен механизм выбора релевантной информации — retrieval. Типичный подход: хранить информацию в векторной базе данных, преобразовывать текущий запрос в вектор и извлекать наиболее семантически близкие записи. Это позволяет агенту «вспоминать» релевантную информацию даже из очень большого архива.

## Planning: от реактивности к проактивности

Планирование — это то, что отличает агента, реагирующего на ситуации, от агента, стратегически идущего к цели. Без планирования агент оптимизирует каждый шаг локально, что далеко не всегда приводит к глобальному оптимуму. С планированием агент видит задачу целиком, понимает зависимости между шагами и может предвидеть последствия своих решений.

Планирование в AI-агентах может быть реализовано на разных уровнях сложности.

**Отсутствие явного планирования (reactive)** — агент на каждом шаге выбирает действие, которое кажется наиболее подходящим прямо сейчас. Это простейший подход, который часто работает для несложных задач, но плохо масштабируется на задачи, требующие координации множества шагов.

**Планирование с фиксированной декомпозицией (decomposition)** — агент в начале разбивает задачу на подзадачи и затем последовательно их выполняет. Это работает, когда структура задачи известна заранее и не меняется по ходу выполнения.

**Адаптивное планирование (adaptive)** — агент формирует начальный план, но готов его корректировать на основе результатов выполнения. Если какой-то шаг провалился или выявилась новая информация, план обновляется.

**Иерархическое планирование (hierarchical)** — задачи разбиваются на уровни абстракции. Верхний уровень определяет стратегию, нижние уровни — конкретные тактики. Это позволяет справляться с очень сложными задачами, но требует существенных когнитивных ресурсов от модели.

На практике планирование в LLM-агентах обычно реализуется через специальные промпты, побуждающие модель сначала обдумать задачу, составить план, а затем следовать ему. Паттерн «Plan-and-Execute», который мы подробно рассмотрим в модуле о LangGraph, формализует этот подход: один агент (планировщик) создаёт план, другой агент (исполнитель) выполняет шаги плана.

## Среда (Environment): арена действий агента

До сих пор мы говорили о внутренних компонентах агента. Но агент существует не в вакууме — он действует в среде, которая определяет контекст его работы. Понимание свойств среды критически важно для проектирования агентов.

**Полнота наблюдаемости.** В полностью наблюдаемой среде агент имеет доступ ко всей информации, необходимой для принятия решений. В частично наблюдаемой среде часть информации скрыта, и агент должен действовать в условиях неопределённости. Большинство реальных сред частично наблюдаемы: агент, анализирующий документы, не знает, какие документы ещё существуют; агент, работающий с API, не видит внутреннее состояние сервиса.

**Детерминированность.** В детерминированной среде одно и то же действие всегда приводит к одному результату. В стохастической среде результат действия непредсказуем. API может вернуть ошибку из-за временных проблем сети; поисковый запрос может вернуть разные результаты в зависимости от персонализации.

**Статичность.** Статичная среда не меняется, пока агент «думает». Динамичная среда продолжает эволюционировать. Веб-страница может обновиться между тем, как агент её загрузил, и тем, как он решил на неё сослаться.

**Дискретность.** Дискретная среда имеет конечное число состояний и действий. Непрерывная среда — бесконечное. Для LLM-агентов среда обычно дискретизируется через конечный набор инструментов, но пространство возможных промптов, по сути, непрерывно.

**Эпизодичность.** В эпизодической среде каждый эпизод независим от предыдущих. В последовательной среде текущий эпизод зависит от прошлых. Большинство агентных задач последовательны: результат предыдущего действия влияет на доступные опции и оптимальную стратегию.

Проектируя агента, важно честно оценить свойства среды, в которой он будет работать. Агент, спроектированный для полностью наблюдаемой детерминированной среды, будет плохо работать в частично наблюдаемой стохастической среде — и наоборот.

## Цикл работы агента: собираем всё вместе

Теперь мы можем описать полный цикл работы агента, интегрируя все рассмотренные компоненты.

**Шаг 1: Получение входных данных.** Агент получает задачу от пользователя (или от другого агента, или по расписанию). Входные данные помещаются в рабочую память.

**Шаг 2: Формирование контекста.** Система извлекает релевантную информацию из долгосрочной памяти: историю предыдущих взаимодействий, связанные документы, примеры решений похожих задач. Эта информация добавляется к рабочей памяти.

**Шаг 3: Планирование (опционально).** Если задача сложная, агент формирует план её выполнения. План может быть явным (список шагов) или неявным (общая стратегия в голове модели).

**Шаг 4: Рассуждение.** LLM анализирует задачу с учётом контекста и плана. Модель может «думать вслух» (chain of thought), выявляя релевантные факты, формулируя гипотезы, идентифицируя пробелы в информации.

**Шаг 5: Принятие решения.** На основе рассуждений модель решает, что делать дальше. Вариантов три: вызвать инструмент для получения информации или выполнения действия; дать финальный ответ пользователю; запросить уточнение у пользователя.

**Шаг 6: Выполнение действия.** Если выбран инструмент, система вызывает его с параметрами, сгенерированными моделью. Результат возвращается агенту.

**Шаг 7: Обновление памяти.** Действие и его результат записываются в память. План может быть скорректирован на основе новой информации.

**Шаг 8: Проверка условия завершения.** Система проверяет, достигнута ли цель, превышен ли лимит шагов, возникла ли неустранимая ошибка. Если нет — возврат к шагу 4.

Этот цикл может показаться простым на бумаге, но его качественная реализация требует решения множества инженерных задач: обработка ошибок на каждом шаге, балансировка между скоростью и качеством, предотвращение зацикливания, обеспечение детерминированности где возможно.

## Визуализация архитектуры: диаграмма взаимодействия

Для наглядности представим архитектуру агента в виде схемы. Центральный узел — LLM, который связан со всеми остальными компонентами. От пользователя к LLM идёт стрелка «Задача», от LLM к пользователю — «Ответ». LLM соединён двунаправленными стрелками с блоками «Инструменты» (вызов → результат), «Память» (запрос → данные, новые данные → сохранение) и «Планировщик» (запрос плана → план, обновление плана). Все компоненты находятся внутри границы «Среда», которая представляет внешний мир.

```
┌────────────────────────────────────────────────┐
│                     СРЕДА                      │
│ ┌────────────────────────────────────────────┐ │
│ │                   АГЕНТ                    │ │
│ │                                            │ │
│ │ ┌───────────┐      ┌─────────────────────┐ │ │
│ │ │  Память   │◄────►│                     │ │ │
│ │ └───────────┘      │                     │ │ │
│ │                    │        LLM          │ │ │
│ │ ┌───────────┐      │       (мозг)        │ │ │
│ │ │Планировщик│◄────►│                     │ │ │
│ │ └───────────┘      │                     │ │ │
│ │                    └──────────┬──────────┘ │ │
│ │                               │            │ │
│ │                    ┌──────────▼──────────┐ │ │
│ │                    │    Инструменты      │ │ │
│ │                    │  (API, файлы, код)  │ │ │
│ │                    └─────────────────────┘ │ │
│ └────────────────────────────────────────────┘ │
│                          ▲                     │
│                          │                     │
│                   ┌──────┴─────┐               │
│                   │Пользователь│               │
│                   └────────────┘               │
└────────────────────────────────────────────────┘
```

Эта схема — упрощение, но она отражает ключевые потоки данных. В реальных системах добавляются дополнительные компоненты: оркестратор, управляющий циклом; модуль безопасности, фильтрующий опасные действия; система мониторинга, собирающая метрики. Но базовая структура остаётся той же.
